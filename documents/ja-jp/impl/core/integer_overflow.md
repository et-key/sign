# Sign 整数オーバーフロー仕様

## 概要

C における最大の設計上の欠陥のひとつが **signed 整数オーバーフローの未定義動作（UB）** である。
Sign はリテラルの型によってオーバーフロー挙動を**静的に確定させる**ことで、
この問題を構造的に排除する。

---

## 1. Sign の数値リテラル型とオーバーフロー挙動

Sign の数値リテラルはプレフィックスによって型と意味が決まる。

| リテラル | 型の意味 | オーバーフロー挙動 | 理由 |
|---------|---------|-----------------|------|
| `0` | 通常の整数（十進） | **ラップアラウンド** | 算術演算の自然な挙動。システムに悪影響なし |
| `0r00` | ビット列（raw バイナリ）| **ラップアラウンド** | bit演算はラップが前提（暗号・ハッシュ等） |
| `0x00` | アドレス（ポインタ）| **`__`（Unit）に収束** | 不正アドレスへのアクセスを防ぐ |
| `0u00` | Unicode コードポイント | **`__`（Unit）に収束** | 無効コードポイントの伝播を防ぐ |

**原則**:
- 意味的に「ラップしても安全」な型 → ラップアラウンド
- 意味的に「ラップすると危険」な型 → `__`（Unit）に収束

---

## 1.5 layer × リテラル型の切り分け

リテラル型は `layer` 制約とも連携する。
使用できるリテラル型は layer によって異なり、対応しない命令セットを使うとコンパイルエラーになる。

| リテラル | layer: 0 | layer: 1 | layer: 2+ | 理由 |
|---------|---------|---------|----------|------|
| `0`（整数）| ✓ | ✓ | ✓ | 整数演算は全 layer で有効 |
| `0r00`（bit列）| ✓ | ✓ | ✓ | ビット演算は全 layer で有効 |
| `0x00`（アドレス）| ✓ | ✓ | ✓ | アドレス演算は全 layer で有効 |
| `0u00`（Unicode）| ✓ | ✓ | ✓ | バイト列として扱う（layer: 0 では文字列出力不可） |
| `0.0`（float）| ❌ | ❌ | ✓ | FPU は `layer: 2` 以上で有効 |
| SIMD ベクタ型 | ❌ | ❌ | ✓（layer: 3+） | SIMD 命令は `layer: 3` 以上 |

```sign
` layer: 0 で float を使うとコンパイルエラー
x : 3.14    ` ← layer: 2 以上でないとエラー
            `   「FPU が初期化されていないレイヤーで float は使用不可」
```

型リテラルとオーバーフロー挙動が `layer` と組み合わさることで、
「このコードがどのハードウェア状態で動くか」がコンパイル時に静的に保証される。

## 2. C との比較

### C の signed 整数オーバーフロー（UB）

```c
// signed int のオーバーフロー → 未定義動作（UB）
int x = INT_MAX;  // 2147483647
int y = x + 1;    // UB: コンパイラは「起きない」と仮定して最適化を壊せる

// 実際に起きたこと（GCC の最適化による CVE の原因パターン）
for (int i = 0; i <= INT_MAX; i++) {  // コンパイラ: i は INT_MAX を超えない → 無限ループ生成
    // ...
}
```

C の仕様では `signed` 整数のオーバーフローは **UB**。
コンパイラはオーバーフローが「起きない」と仮定して最適化し、
プログラムが予期しない動作をする。これが多くのセキュリティ脆弱性の原因になった。

### C の unsigned 整数オーバーフロー（定義済み）

```c
// unsigned int のオーバーフロー → ラップアラウンド（定義済み動作）
unsigned int u = UINT_MAX;  // 4294967295
unsigned int v = u + 1;     // → 0（確定、定義済み）
```

`unsigned` はラップアラウンドが規格で定義されている。

### Sign の整数オーバーフロー（すべて定義済み）

```sign
` 通常整数（0）- ラップアラウンド（定義済み）
x : 2147483647
y : x + 1           ` → ラップアラウンド（定義済み動作）

` ビット列（0r）- ラップアラウンド（定義済み）
b : 0rFF            ` 8bit: 255
c : b + 0r01        ` → 0r00（ラップ、定義済み）

` アドレス（0x）- Unit に収束（定義済み）
addr : 0xFFFFFFFF
next : addr + 0x01  ` → __（Unit）← アドレス空間の終端を超えた
```

**Sign には「未定義動作」が存在しない。** すべてのオーバーフローは型によって挙動が確定している。

## 3. 暗号・ハッシュでのラップアラウンド活用

`0r` リテラルはビット列（raw binary）であり、ラップアラウンドが設計上の前提：

```sign
` SHA-256 の加算（32bit ラップアラウンド）
add32 : a b ? a + b         ` 0r 型同士の + は自動的にラップ

` AES の GF(2^8) 乗算的なもの
` 0r00 型の演算は暗号実装に安全に使える
```

暗号アルゴリズムは意図的なラップアラウンドを利用するが、
Sign では `0r` 型を使うことで**意図が型に明示される**。
「なぜラップするのか」がコードを読むだけでわかる。

---

## 4. アドレスオーバーフローの安全性

`0x` 型（アドレス）がオーバーフローすると `__`（Unit）に収束する：

```sign
` 配列の末端を超えたアクセス
arr  : 0x1000        ` 配列の先頭アドレス
size : 256
end  : arr + size    ` → 0x1100（正常）

` 意図的に範囲外を計算すると
oob : arr + size + 1  ` → コンパイル時にサイズが既知なら静的エラー
                       ` 動的なら 0x1101 → Unit へ収束（クラッシュしない）
```

C での同等コードはバッファオーバーフローの典型的な原因。
Sign ではアドレス型の演算結果が自動的に `__` に収束し、
後続の演算も `__` を伝播するだけで**クラッシュや不正アクセスが発生しない**。

---

## 5. C との対比まとめ

| 型の種類 | C の挙動 | Sign の挙動 |
|---------|---------|-----------|
| signed 整数オーバーフロー | **UB**（コンパイラが何をしてもよい） | ラップアラウンド（定義済み） |
| unsigned 整数オーバーフロー | ラップアラウンド（定義済み） | ラップアラウンド（定義済み） |
| ポインタ演算オーバーフロー | UB または実装定義 | `__`（Unit）に収束（定義済み） |
| 文字コードオーバーフロー | 実装定義 | `__`（Unit）に収束（定義済み） |

**Sign では全ケースが定義済み**。C の signed UB 問題は構造的に存在しない。
