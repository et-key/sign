## 0. 実装に直結する重要性質（クイックリファレンス）

### 0.1 未定義識別子のUnit収束（予約語の完全排除）

Sign言語には `null`, `nil`, `false` などの予約語は存在しません。
**「現在スコープにおいて定義されていない識別子はすべて `__`（無）として評価される」**というコア仕様が存在します。

#### 仮想キーワードとしての活用
この仕様により、ユーザーは任意の英単語やUnicode記号をゼロコストの「仮想キーワード」として利用できます。
```sign
@lazy tick
@iterator next
```
未定義の識別子（`tick`, `next`）は安全に `__` として評価されるため、可読性を高めるための自然言語的なマーカーとして機能します。

### 0.2 UnitとFalseの同型性と短絡評価

言語組み込みの論理積（`&`）および論理和（`|`）は短絡評価（Short-circuit）を行います。

```sign
 appendの終了条件と再帰ステップ
append : [x ~xs] [y ~ys] ?
    xs & x (append xs (y ys)) | @x y ys
```
- `xs` が空リストではない（真）場合、後続の再帰ステップが評価される
- `xs` が空リスト（`__`／偽）の場合、`|` の右辺（終了条件）が評価される

### 0.3 演算子の安全な無効化

```sign
[+] : ~x ? __
```
可変長引数 `~x` によってオペランドをすべて吸収し、明示的に `__` を返すことで、演算子が用いられた式全体を安全に `__` に崩壊させます。

### 0.4 `__` と `$__` の明確な区別

| | `__`（即時崩壊） | `$__`（遅延サスペンド） |
|:---:|---|---|
| **性質** | 計算パスの「死」 | 計算の「休眠（サスペンド）」 |
| **挙動** | Unit伝播で即座に式全体を崩壊 | `@` で要求されるまで待機 |
| **用途** | 短絡評価・エラー伝播 | 遅延ストリームの終端シグナル |

`$__` は「Unitが配置されているメモリ上のアドレス」を返す有効な値（非Unit）です。`$__` = `__` という等号は成立しません。

## 1. イントロダクション

> [!IMPORTANT]
> **`__` は Sign を圏として定義するために必要な零対象（Zero Object）である。**
> 以下の全ての振る舞いはこの一事実から導出される。

零対象（Zero Object）とは、**初対象（Initial Object）と終対象（Terminal Object）が一致した対象**です。

$$\__ = \text{Initial Object} \cong \text{Terminal Object}$$

この一致により、`__` はあらゆる演算に対して以下の双対的振る舞いを同時に持ちます。

| 対象としての役割 | 演算 | 振る舞い |
|:---:|:---:|---|
| **終対象**（積の吸収元） | `&`, `+`, `*`, ... | `__ op x = __` |
| **初対象**（余積の単位元） | `|`, `;`, ` `, `,` | `__ op x = x` |

これは「論理的に偽として機能する」という付帯的な説明ではなく、圏論的に記述すると自然に現れる性質です。
以下の各セクションは、零対象としての `__` から導出される各性質を詳述します。

## 2. 双代数としての数学的基盤

### 2.1 基本性質（零対象からの導出）

- **零対象**: Signを圏として定義するために必要な、始対象と終対象の一致点（**これが第一原理**）
- **双代数の単位元**: リスト構造（余代数）と関数合成（代数）の両方の単位元（零対象から導出）
- **値としての性質**: `__ = []`（空リストと等価）
- **関数としての性質**: 余積の単位元として「プログラムの流れを素通りする」
- **重要な区別**: `__ ≠ 0`（数値ゼロとは明確に異なる：零対象 ≠ 数値）

### 2.2 双代数構造の定義

Sign言語のリスト構造は双代数 `(List, unit, join, extract, duplicate)` を形成します：

#### 代数構造（モナド）

- `unit : A → List A` where `unit(x) = [x]`
- `join : List(List A) → List A` where `join([[a₁, a₂], [b₁, b₂]]) = [a₁, a₂, b₁, b₂]`

#### 余代数構造（コモナド）

- `extract : List A → A` where `extract([x]) = x`, `extract([]) = __`
- `duplicate : List A → List(List A)` where `duplicate([a, b]) = [[a], [b]]`, `duplicate([]) = [[]]`

### 2.3 圏論的証明：`__`の双代数単位元性

**定理**: `__`は双代数 `(List, unit, join, extract, duplicate)` の単位元である

**証明**:

#### モナド単位元の証明

```Haskell

__ >>= f = __                // left unit law (空リストなので結果も空リスト__になる)
m >>= (\x → __) = __         // right unit law

具体例：
__ >>= [+ 2] = [+ 2](__) = __
[1,2,3] >>= (\x → __) = __

```

#### コモナド単位元の証明

```haskell

extract(__) = extract([]) = __           // extraction law
duplicate(__) = duplicate([]) = [[]] = [__]   // duplication law

```

#### 双代数互換条件の検証

```haskell

extract ∘ unit = id:
extract(unit(__)) = extract([__]) = __ = id(__) ✓

duplicate ∘ unit = unit ∘ unit:
duplicate(unit(__)) = duplicate([__]) = [[__]]
unit(unit(__)) = unit([__]) = [[__]] ✓
```

## 3. Unit演算の完全仕様

### 3.1 関数文脈での演算

#### 3.1.1 算術演算子との相互作用

```Sign

`関数の単位元として部分適用を生成
`吸収元（墓標・即時崩壊）として機能し、全体を __ に崩壊させる

X + __ → __

`【比較】Hole（_）を使用した場合は、部分適用（プレースホルダによる関数化）を生成する

` Unit は非圏論的演算子（算術など）において、左辺では「吸収元」、右辺では「単位元」として振る舞う
```

#### 3.1.2 比較演算子との相互作用

```Sign

`大小比較や等値比較における吸収（比較演算子は両辺とも吸収元）
`左辺Unit: 比較対象が存在しない → 演算不成立 → __
__ < X → __       X < __ → __

__ >= X → __      X >= __ → __

`【比較】Hole（_）を使用した場合は、部分適用（比較関数）を生成する

_ = X → [= X]     X = _ → [X =]

```

> [!NOTE]
> **比較演算子と Unit の相互作用**
>
> **左辺が `__` の場合:** `__` は値として確定していないため、比較演算子は演算自体が成立しない。よって `__`（吸収元）が返る。
>
> **右辺が `__` の場合:** 同様に比較対象が存在しないため演算が成立しない。よって `__`（吸収元）が返る。
> これは算術演算子（`x + __ = x`）の「右辺Unit=単位元」とは異なる。比較演算子は**値の順序・等値関係の判定**であり、比較対象の欠損は演算の失敗を意味する。
>
> **`X != __` が唯一の例外:** `X` が非 Unit である確定した値を持つことが前提であり、「XはUnitではない」という比較は論理的に真。よって真の場合の返値ルール（LHS を返す）が適用され、`X` が返る。
>
> なお、`3 < 2 != 4` のような異種比較の連接は**構文上 NG**（比較演算子の連接は同種のみ許可）。
> `3 < 2` の結果を `!=` で使いたい場合は `(3 < 2) | 4` のように余積演算子を使う。

明示的にUnit（`__`）を引数として渡した場合の挙動は、対象となる関数のアリティ（引数の満たされ具合）によって双代数的な役割を明確に変えます。

```Sign
` 例外： 前置 @ によるデリファレンスは、右辺が飽和時に評価を駆動する
```

> [!IMPORTANT]
> **完全性公理** (`Type_System_ja-jp.md` §3.4)：`∀f, f __ = __`
>

> `__` を引数として渡すことは「評価しきれない」（偽）を意味するため、全体が `__` に消去する。
> 前置 `@` によるデリファレンス（評価の駆動）はこの公理の外側にある別概念である。

### 3.3 積・余積における零対象としての振る舞い

`__` は**零対象（Zero Object）**として、積（`&`）と余積（`|`）でそれぞれ異なる役割を担います。これは「論理的に偽として機能する」という付帯的な性質ではなく、**零対象の定義から必然的に導出される振る舞い**です。

> [!IMPORTANT]
> 零対象とは**初対象（Initial Object）と終対象（Terminal Object）が一致したもの**です。
> これにより `__` は積においては吸収元、余積においては単位元として、同時に機能します。

#### 3.3.1 論理積 `&` ── 積（Product）における零対象

`&` は圏論的な積（Product）演算です。零対象 `__` は積の**吸収元（零元）**として機能します。

```sign
__ & X → __    ` 終対象として積に吸収（0 × x = 0）
X & __ → __    ` 同上
```

#### 3.3.2 論理和 `|` ── 余積（Coproduct）における零対象

`|` は圏論的な余積（Coproduct）演算です。零対象 `__` は余積の**単位元**として機能します。

```sign
__ | X → X    ` 初対象として余積の単位元（0 + x = x）
X | __ → X    ` 左辺が非Unitなら短絡評価
```

#### 3.3.3 排他的論理和 `;` ── 余積の差分

`;` は余積における対称差（Symmetric Difference）です。`__` は余積の単位元として同様に機能します。

```sign
__ ; X → X    ` 余積の単位元
X ; __ → X    ` 同上
```

#### 3.3.4 否定 `!`

> [!IMPORTANT]
> **前提：ラムダが Unit と同型になる条件**
>
> Sign において「未評価なラムダは Unit と同型」である。
> これは、一般のラムダが副作用を含む可能性を持ち、Id 射の観点から**評価予定が静的に確定できない**ことに由来する。
> 評価予定が未定 = 「何を返すか約束できない」= Unit と同型。
>
> **例外：評価予定が確定したラムダ**
>
> `!__` が返す Id 射は**副作用のない純粋な恒等関数**であり、Id 射の観点から評価予定が静的に確定している。
> よって「未評価だが Unit ではない」特別なラムダとして機能し、真（非 Unit）として扱われる。
>
> 注：`!__ __` は理論的に正しく `__` を返す（Id射に Unit を渡すため）。
> しかし `!__` 自体は評価予定確定ラムダであるため `__` ではない。

```sign
!__    ` → Id射（評価予定確定・非Unit・真に相当）
!expr  ` → __  （expr が非Unit の場合）
```

> [!NOTE]
> **外延性公理による `!__` とブラケット表記の恒等射の同一性証明**
>
> `!__` は「副作用のない純粋な恒等関数」として定義されている以上、任意の $x$ に対して `!__ x = x` が成り立つ。一方、ブラケット記法の真の恒等射 `[__ ]` / `[ __]` も静的脱糖 `$p0 ? $p0` の定義より任意の $x$ で `[__ ] x = x` が成り立つ。`Kan_Extensions_Unification_ja-jp.md` §3.5.2 が `__ ≠ 0` の証明に用いた関数の外延性公理 $A = B \iff \forall x (A \text{ op } x = B \text{ op } x)$ をここに適用すると：
>
> ```
> !__ x = x（定義より）
> [__ ] x = x（静的脱糖 $p0 ? $p0 の定義より）
> ∴ 外延性公理により !__ = [__ ] = [ __]
> ```
>
> さらに、裸のHole `_`（何とも演算子結合していない単体の `_`）の静的脱糖極限も同じ形 `$p0 ? $p0` に収束する（`Hole_Static_Desugaring_Spec_ja-jp.md` 参照）。したがって `!__`・`[__ ]`・`[ __]`・裸の `_` の静的脱糖極限は、いずれも**同一の恒等射に到達する4つの異なる構文経路**である。
>
> ただし `_` 自体はランタイム値ではなくコンパイル時に完全に消える構文糖衣（静的脱糖）であり、`!__` はランタイムに存在する値である点で両者は同じ圈に属さない。同一性が成立するのはあくまで「`_` を静的脱糖した後の項」と `!__` の間である。

### 3.4 アドレス・入出力演算

```Sign

`Unit自身への参照
` ⚠ 例外：$__ は __ を返さない
` $__ は「Unit が配置されているメモリ上のアドレス」を返す有効な参照（非 Unit）。
` 完全性公理（f __ = __）の適用対象外。アドレス演算子 $ の意味論が優先される。
$__ → <Unitのアドレス>  ` 非Unit（有効なアドレス値）

`Unitからの入力は吸収
@__ → __

`Unitへの出力は無効化（/dev/null相当）
__ # X → __

```

> [!IMPORTANT]
> **`$__` は例外的に `__` を返さない。**
>
> `$` 演算子は「そのアドレスを取得する」操作であり、`__` が引数であっても
> 演算子の意味論（アドレス取得）が完全性公理より優先される。
> `$__` の結果は Unit のメモリアドレスを表す有効な値（Address 型）であり、
> これを `__` に収束させると「アドレスを持つ何か」と「無」の区別が失われる。
> 逆に `@__`（Unit からの読み出し）は「何もない場所からの読み出し」として `__` に吸収される。

## 4. 双代数の分配法則

Sign言語の演算子関数化は双代数の分配法則として表現されます：

```Sign

`分配法則: (f ⊗ g)(hole(x)) = hole(f(x)) ⊗ hole(g(x))
(+ ⊗ *)(hole(x)) = hole(+(x)) ⊗ hole(*(x))

具体例:
_ + 3 → [+ 3]    `Holeによる部分適用の生成
_ * 5 → [* 5]    `hole(*(5))を生成

```

この分配法則により、演算子の自然な関数化と部分適用が双代数の構造から導かれます。

## 5. ARM64上での最適実装

### 5.1 Unit値の表現

- **NULL pointer使用**: Unit値をNULLポインタとして表現
- **条件付き命令活用**: AArch64の条件付き命令でUnit判定を効率化
- **レジスタ最適化**: xZRレジスタの特性を活用

### 5.2 Unit値のバイナリ表現（`\x00`）

> [!IMPORTANT]
> **`\x00`（ヌルバイト）はSignにおけるUnitのバイナリ表現である。**
>
> **⚠️ `0x00`（アドレス値ゼロ）とは全く異なる概念である。混同しないこと。**

Signのパーサー（およびセルフホストコンパイラ）では、入力ストリーム中の生の `\x00` バイトは**文字データ（Char型）としてでも、アドレス値としてでもなく、Unit型の値として直接解釈される**。

> [!CAUTION]
> **`\x00`（ヌルバイト）と `0x00`（アドレスリテラル）の区別**
>
> | 表記 | 種別 | 意味 |
> |---|---|---|
> | `\x00`（入力ストリーム中の生バイト） | **型シグナル** | Unit型そのもの。**値を持たない。** |
> | `0x00`（Signソース中のリテラル） | **Address型の値** | アドレス0番地。**有効な値を持つ。** |
>
> `\x00` はアドレス空間上の「0番地」を指すポインタではない。
> それは「値が存在しない」という型状態そのものを表す入力ストリームの信号である。

| ソース表現 | 解釈 | 用途 |
|---|---|---|
| `__` | Unit値 | 人間が書くソースコード |
| `\x00`（生バイト） | Unit型として直接識別 | 機械生成トークン・セルフホスト時のワイヤフォーマット |
| `0x00` | **Address型の値ゼロ** | 通常のアドレスリテラル（Unitとは無関係） |

この設計により：

1. **セルフホスト時の効率化**: SignコンパイラがUnit値をトークン列として出力する際、`__`（2バイト）ではなく `\x00`（1バイト）として emit することで処理を効率化できる。
2. **型シグナルとしての明確性**: `\x00` バイトは文字リテラル（`\a`、`\n` 等）とは構文上区別され、文字として扱われることがない。Signの文字リテラルはバックスラッシュ+1文字（`\\x`）の形式であるため、生の `\x00` バイトとは衝突しない。
3. **Stringへの包含禁止**: `\x00` はString型に含めることができない。Signにおいて `\x00` は常にUnitを意味し、文字データとしての意味を持たない。

```
入力バイト \x00  →  lex → Unit型トークン（= __）
                         ↓
           パーサーが unit ルールでキャッチ
                         ↓
           ASTノード: { type: "Atom", dataType: "unit", value: "__" }

※ 入力ストリーム中の "0x00" テキスト → address型トークン（値ゼロのアドレス）← 別物！
```

### 5.3 双代数演算の最適化

## 7. 設計原則と理論的意義

### 7.1 双代数単位元としての一貫性

1. **代数的一貫性**: モナド法則を満たす関数合成の単位元
2. **余代数的一貫性**: コモナド法則を満たすリスト構造の単位元
3. **双代数互換性**: 代数と余代数の相互作用を保持
4. **自然変換性**: 関数とリストの間の自然な変換を提供

### 7.2 実装効率との統合

- **理論的純粋性**: 圏論的に記述・検証できる設計
- **実装効率**: ARM64での高効率な機械語生成
- **型安全性**: コンパイル時の双代数構造検証
- **最適化可能性**: 双代数の性質を活用した自動最適化

## 8. 結論

Sign言語のUnit (`__`) は、単なる「便利な記号」ではなく、評価規則の第一原理から生まれた言語の基盤要素です。圏論的に記述すれば双代数の単位元と同型な振る舞いを示し、モナドとコモナドの両方の観点から整合的に説明できます。これにより、関数型プログラミングとリスト処理の統一的な表現と、「見えない強い型付け」および「ゼロコスト抽象化」の根拠を与えています。

圏論的に見れば双代数的な構造を持つこの設計は、実装効率と理論的整合性を両立し、プログラマにとって直感的な言語として機能します。
