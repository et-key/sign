# Sign ビルドシステム仕様：`option.ms` とコンパイル階層

## 1. 概要

> [!IMPORTANT]
> **Sign のビルド設定は、コマンドライン引数ではなく `option.ms`（MetaObjectForSign）ファイルによって管理される。**
> これはSign言語自身のデータ記述能力を使ってコンパイラを設定するという、セルフホスト哲学の自然な帰結である。

Sign のコンパイラ設定は以下の原則に従う：
1. **設定はファイルで宣言する** — コマンドライン引数は緊急時のオーバーライドのみ
2. **フォルダ単位で有効** — 設定を変更したいディレクトリにのみ `option.ms` を配置する
3. **レキシカルな継承** — 子フォルダは親の設定を暗黙的に継承し、必要な部分のみ上書きする
4. **動的追加は許可しない** — `option.ms` の構造はスキーマで静的に決定される

---

## 2. `ms` フォーマット（MetaObjectForSign）

`ms` は Sign の積型（Product）記法をそのままデータ記述に使ったフォーマットである。

> [!IMPORTANT]
> **ファイル名によるメタとデータの静的区分**
> Node.jsが `package.json` や `.mjs/.cjs` のようにファイルシステム境界で解釈を分けたのと同様に、Signもコンパイル時メタデータと実行時コード/データを静的に分離する。
> - **`*.ms`** (MetaObjectForSign)：コンパイラがビルド時・コンパイル時にのみ読み取るメタ定義ファイル。
> - **`*.sn`** (Sign)：ランタイム（実行時）に実行されるコード、および実行時にロードされるプログラムデータ。
> 
> この静的区分により、実行時プログラムがコンパイラ専用のメタ情報に誤アクセスする等の混乱を構造的に排除し、ツールチェーンの解析精度と安全性を最大化する。

| JSON | ms |
|------|-----|
| `"key": value` | `key : value` |
| `"string"` 必須 | 識別子はそのまま、文字列はバッククォート |
| `null` / `false` | `__`（Unit に収束） |
| コメント不可（標準） | 行頭バッククォート `` ` `` でコメント記述 |
| 動的追加可能 | スキーマ固定・静的のみ |
| `[1, 2, 3]` | `1 2 3` または `1,2,3` など、Sign の積、余積の仕様に従う |

## 3. フォルダごとの `option.ms` 継承モデル

### 3.1 ディレクトリ構造
変更がないフォルダ（例: `drivers/`）には `option.ms` を置く必要はなく、自動的に親ディレクトリの設定を継承する。

```
project/
  option.ms              ← ルート設定（全体のデフォルト）
  │
  ├── kernel/
  │     option.ms        ← layer: 0 に変更あり → 要る
  │     ├── drivers/
  │     │    └── uart.sn   ← option.msなし（kernel/の設定をそのまま継承）
  │     └── mm/
  │           option.ms  ← optimize: 3 へ変更あり → 要る
  │
  ├── userspace/
  │     option.ms        ← layer: 2 に戻す変更あり → 要る
  │
  └── lib/
        option.ms        ← target: wasm など別ターゲットへ変更あり → 要る
```

### 3.2 継承ルール
- 対象ファイルのフォルダからルートに向かって `option.ms` を探索し、スタックして適用する
- 近いフォルダで定義されたキーほど優先（上書き）される
- `inherit : false` を宣言した場合、親の設定を無視して独立する（デフォルト: true）

## 4. コンパイル階層（Layer）とリンク設定

`layer` フィールドは数値で環境の抽象度を指定する。
`layer` は単なるビルド設定ではなく、**コンパイル時の使用可能機能セットの宣言**である。
コンパイラは `layer` の値に基づいて、禁止された機能の使用をコンパイルエラーとして報告する。

**各 layer は下位 layer を包含する（累積的に機能が追加される）。**

| 数値 | 別名 | 説明 | alloca | Int/UInt | Float/Double | Vector (SIMD) | 動的モジュール | 対応するリンク |
|------|------|------|--------|----------|--------------|---------------|----------------|----------------|
| `0` | bare | RAMがまだ初期化されていない最低レイヤー。BIOS/UEFIの初期フェーズ相当。`@`/`#` は自動 volatile。 | ✗ | ✓ | ✗ | ✗ | ✗ | `link: static` のみ |
| `1` | alloc | RAMが開通し alloca が使用可能になったレイヤー。カリー化ABIや `Rc`/`Arc` が有効。FPUはまだ不可。 | ✓ | ✓ | ✗ | ✗ | ✗ | `link: static` のみ |
| `2` | fpu | FPU（浮動小数点演算器）が初期化済み。`Float`/`Double` 型が使用可能になる。 | ✓ | ✓ | ✓ | ✗ | ✗ | `link: static` のみ |
| `3` | simd | SIMD（SSE/AVX/NEON等）が初期化済み。`Vector` 型が使用可能になる。ストライドアクセスのVECTOR化も可。 | ✓ | ✓ | ✓ | ✓ | ✗ | `link: static` のみ |
| `4` | std | OS依存のフル機能層。動的モジュール解決が有効。 | ✓ | ✓ | ✓ | ✓ | ✓ | `link: dynamic` / `static` |

> [!NOTE]
> `layer` を指定しない場合、デフォルトは `4`（std）となる。
> 各レイヤーにおけるメモリ操作やモジュール解決の数学的・意味論的詳細については、[システムアーキテクチャ仕様書](System_Architecture_ja-jp.md) を参照のこと。

> [!IMPORTANT]
> **型と layer の対応**
> `3.14` のような float リテラルを `layer < 2` のコード中で使用するとコンパイルエラーになる。
> SIMD型の適用も `layer < 3` では同様にエラー。
> これにより、未初期化のハードウェア機能へのアクセスを構造的に防止する。

### 4.1 `option.ms` と `.sn` の共同作業モデル

BIOS/UEFIレベルの開発では、`option.ms` と `.sn` コードが役割を分担して協働する：

- **`option.ms`**：メモリマップ・エントリポイント・layer制約などを**宣言**する
- **`.sn` コード（`layer: 0`）**：RAM初期化・スタック設定などの**実際の処理**を記述する

起動シーケンスと layer の対応例（x86）：

```sign
` boot.sn — layer: 0 のフェーズ（RAMがまだない）
` Int/UInt のみ使用可能。Float/SIMDはコンパイルエラー

ddr_init :
    0xFED10000 # 0x12345678   ` メモリコントローラ設定
    0xFED10004 # 0x87654321   ` タイミングパラメータ

fpu_init :
    ` CR0.EMをクリア、CR0.MPをセット（アドレス操作のみ）
    0xCR0 # (@ 0xCR0 & 0xFFFFFFFB | 0x00000002)

simd_init :
    ` CR4.OSFXSR、CR4.OSXMMEXCPTをセット
    0xCR4 # (@ 0xCR4 | 0x00000600)

_start :
    ddr_init    ` → layer 1（alloca有効）相当
    fpu_init    ` → layer 2（Float有効）相当
    simd_init   ` → layer 3（SIMD有効）相当
    ` 以降は全型が使用可能
```

`layer: 0` コード内で alloca / Float / SIMD 型を使用するとコンパイルエラーになる。
この制約により、未初期化のハードウェア機能へのアクセスを構造的に防止する。

## 5. `option.ms` スキーマ全フィールド一覧

```son
` option.ms — 完全フィールド一覧（すべてオプション）

` ─── ターゲット ───────────────────────────────────────────
target   : `rust`    ` rust | wasm | js | x86_64 | aarch64 | riscv64 | wasm32 | wasm64（デフォルト: rust）
layer    : 4         ` 0=bare | 1=alloc | 2=fpu | 3=simd | 4=std（デフォルト: 4）

` ─── ビルド ────────────────────────────────────────────────
entry    : `main.sn` ` エントリポイント関数名（省略時は main.sn のトップレベル式を実行）
output   : `app`     ` 出力ファイル名（拡張子はtargetに応じて自動）
optimize : 2         ` 0〜3（デフォルト: 0）
                     ` ※ LTO、デバッグ情報(strip)等は optimize レベルから自動導出されるため指定不要

` ─── 継承 ──────────────────────────────────────────────────
inherit  : true      ` false にすると親option.msを無視（デフォルト: true）

` ─── リンク戦略 ─────────────────────────────────────────────
` デフォルトは dynamic（OSのダイナミックローダーに委ねる。Node.js相当）。
` static 時のみ、物理/仮想メモリ配置のアドレス範囲をサブブロックで明示する。
link     : dynamic   ` dynamic | static (以下のサブブロックを設定)

` static リンク時の手動メモリマップ設定（例: 組み込みマイコン）
` link :
` 	static :
` 		memory :
` 			rom : origin 0x08000000  length 1024K
` 			ram : origin 0x20000000  length 128K
` 
` static リンク時の実行時マップ取得設定（例: PC/SBCのOS開発）
` link :
` 	static :
` 		memory :
` 			rom : origin 0x00000000  length 4M
` 			ram : auto
```

## 6. ファイル名とエントリーポイント規約

### 拡張子
Signのソースファイルの正式拡張子は **`.sn`** である。
（互換性のためにコンパイラは内部的に `.sign` も認識するが、新規作成時は `.sn` を推奨する）

### エントリーポイント
ディレクトリビルド時のエントリーポイントファイル名は **`main.sn`** である。
`sign build .` を実行すると、対象ディレクトリの `main.sn` が自動的に起点として認識される。

---

## 7. コンパイルコマンド

```sh
sign build .             ` カレントディレクトリからビルド（main.sn を起点）
sign build kernel/       ` 指定フォルダ以下をビルド
sign build src/main.sn   ` 特定ファイルを指定してビルド（最近傍の option.ms を適用）
```

コマンドライン引数によるオーバーライド（一時的なビルド構成の上書き）：
```sh
sign build . --layer 0 --optimize 0
```
