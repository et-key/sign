# Sign言語 リストモデル・型変換・多次元配列 仕様書

## 1. 基本原則：二つの演算子だけで全リスト操作を表現する

Sign言語のリスト操作は、たった二つの演算子の組み合わせで完結する。

| 演算子 | 名称 | 圏論的意味 | 直感的意味 |
|--------|------|-----------|-----------|
| ` `（スペース） | 余積（coproduct） | 並置・連接 | 「並べる」 |
| `,`（カンマ） | 直積（product） | 構造的組み立て | 「組み合わせる」 |

この二つは圏論における**随伴関手**の関係にある。すなわち、余積と直積は数学的に双対な存在として設計されており、一方が定まれば他方も自然に決まる。

---

## 2. 余積（スペース）：1次元リストの構築と結合

### 2.1 スカラー値の並置

```sign

` カンマ区切り（スカラー型の場合、カンマは省略可能）
1, 2, 3, 4, 5

` スペースで並べるても1次元リストになる
1 2 3 4 5

` 以下の三つは全て等価
myPairs  : 1 2 3 4 5
myPairs0 : [,] 1 2 3 4 5
myPairs1 : 1, 2, 3, 4, 5

myPairs = myPairs0 = myPairs1
```

### 2.2 リストの結合

```sign
` スペースによるリスト結合（余積）= フラットに連接される
[1 2] [3 4] = 1 2 3 4

` フラットなリストになる
1 , 2 3 , 4 = 1 2 3 4
```

---

## 3. 直積（カンマ）：次元の構築

### 3.1 カンマは「次元を上げる」演算子

スペースが「同じ次元に並べる」のに対し、カンマは「新たな次元として組み合わせる」。

```sign
` これは1次元リスト（余積：並べるだけ）
1 2 3 4

` これは2次元配列（直積：次元を上げる）
1 2 , 3 4

` 同じ意味をブロック記法で書いた場合
s :
	1 2 ,
	3 4
```

### 3.2 n次元配列への自然な拡張

カンマの入れ子は、自然にn次元配列に対応する。

```sign
` 2次元（2×3）
1 2 3 , 4 5 6

` 3次元（2×2×2）
[1 2 , 3 4] , [5 6 , 7 8]

` 3次元（2×2×2）ブロック記述の場合
v3 :
	1 2 , 3 4 ,
	5 6 , 7 8

` リストのコピーと持ち上げ（乗算との対応）
[1 2 3 4] * 2 = 1 2 3 4 , 1 2 3 4

` リストの分割（除算との対応）
[1 2 3 4] / 2 = 1 2 , 3 4

` 展開演算子：内側を1段階展開し、その後余積（スペース）が自動評価される
` 内側が1段階展開 → 直積が解かれて余積になる（要、構文検討！）
[1 2 , 3 4]~   = 1 2 3 4
```

`~`（後置）は**内側を1段階だけ展開する**演算子である。展開後、スペース（余積）が自動的に評価されることで、結果が1次元リストになる。「外側のカッコを剥がす」のではなく「内側の構造を一段ほぐす」という理解が正確である。

### 3.3 余積と直積の対称性

```sign
` 余積：フラットな結合
[1 2] [3 4]   = 1 2 3 4

` 直積：構造を保った組み立て
1 2 , 3 4     = [[1 2] [3 4]]

` 展開すれば余積と同じになる（双対性の証明）
[1 2 , 3 4]~  = 1 2 3 4
```

## 4. 型変換の左辺優先規則

### 4.1 基本原則

Sign言語の二項演算において、**左辺の型が演算全体の型を決定する**。右辺は左辺の型に自動変換される。変換できない場合はコンパイルエラーとなる。

この設計の根拠は「二項演算の右辺は、動詞（演算）の目的語として左辺に掛かる」という考え方にある。

```sign
` 左辺が数値 → 右辺の文字列を数値として解釈
0 + `123` = 123

` 左辺が文字列 → 右辺の数値を文字として解釈
`123` + 0 = 0x31 0x32 0x33
```

### 4.2 演算子の種類で挙動が決まる：余積 vs 算術演算子

型変換の種類は「左辺の型」と「演算子の種類（余積か算術か）」の組み合わせで決定される。

**スペース（余積）：構造の連接**
```sign
` 左辺が文字列 → 数値を文字列化して連結
`123` 0 = `1230`

` 左辺が数値 → 単なるタプルに
1 2 `34` = 1 2 `34`
```

**算術演算子（`+` `-` `*` `/` など）：左辺が文字列のとき、各文字をコードポイントとして要素ごとに演算**
```sign
` 加算の単位元（+0）でコードポイントを露出
`123` + 0 = 0x11 0x12 0x13

` 乗算の単位元（*1）でも同じ結果 ← +0 と *1 は等価
`123` * 1 = 0x11 0x12 0x13

` 単位元以外の値で演算することも可能
`123` + 1 = 0x12 0x13 0x14   ` 各コードポイントに+1
`123` * 2 = 0x22 0x24 0x26   ` 各コードポイントを2倍
```

**算術演算子：左辺が数値のとき**
```sign
` 右辺の文字列を数値としてパース
0 + `123` = 123
```

ここで重要なのは `"123" + 0` と `"123" * 1` が等価であるという事実である。`+0` は加算の単位元、`*1` は乗算の単位元——どちらも「数値的に何もしない操作」だが、**算術演算子を文字列に適用する行為そのもの**がコードポイントを露出させる。スペース（余積）だけが唯一の例外として「文字列のまま扱う」演算子である。

### 4.3 リスト上の型変換

```sign
` intリストへの文字列追加（余積）→ 文字列全体が数値として解釈される
1 2 `34` = 1 2 34

` 文字列への数値リスト追加（余積）→ 数値がそれぞれ文字列表現として連結される
`hello` [32 119 111 114 108 100] = `hello32 119 111 114 108 100`
```

### 4.4 文字型の本質が現れる瞬間

`+` 演算子が文字列の左辺に当たると、型抽象化の層が剥がれ、**各文字のハードウェアレベルでの実体（アドレス型 `0x`）** が露出する。

```sign
` 文字列は文字のリスト
`abc` = \a \b \c

` 算術演算子で文字列の本質が現れる
`123` + 0 = 0x11 0x12 0x13
`123` - 0 = 0x11 0x12 0x13
`123` * 1 = 0x11 0x12 0x13
`123` / 1 = 0x11 0x12 0x13

` スペース（余積）では文字列のまま連結される
`123` 123 = `123123`
```

これはSign言語が**型と言語そのものを抽象化している**という設計哲学の自然な帰結である。演算子の意味が型変換の方向を決め、プログラマーは型宣言なしに直感的な操作を行える。

## 5. 混在型（heterogeneous）リストについて

### 5.1 設計上の問いかけ

異なる型を持つ要素を直積で組み合わせることは、構文上は可能である。

```sign
` 数値と文字列の混在タプル
1, `a`, 2, `b`
```

しかし、**このようなタプルが実際に意味のある場面はどれほどあるか？** という問いを立てることが重要である。

### 5.2 設計指針

Sign言語のリストモデルは「同一型の値の集合」を基本とする。混在型が必要な場面では、以下のように辞書型（key-value構造）を使うことで、より明確な意味を持たせることができる。

```sign
` 混在型リストより、辞書型の方が意図が明確
person :
	name : `Alice`
	age  : 30
	score : 95.5
```

### 5.3 型推論との関係

混在型リストに対して演算を行う場合、左辺優先規則に従い先頭要素の型が基準となる。これにより「混在型リストをそのまま扱う」よりも「辞書型で明示的に構造化する」ことが自然に促される。

---

## 6. まとめ：リストモデルの統一性

Sign言語のリストモデルは、以下の一貫した原則から成り立つ。

```
スペース（余積）  ＝  並べる  ＝  1次元での連接
カンマ（直積）    ＝  組む    ＝  次元を上げる
左辺優先規則      ＝  型は演算の文脈から自明に定まる
文字はアドレス型  ＝  型抽象化の層を剥がすと、ハードウェアの姿が現れる
```

これらはすべて「型と言語を抽象化した」という設計哲学の表れであり、プログラマーが型を意識することなく、数学的に一貫した操作を行える基盤を提供している。
