# Sign言語 設計モチベーション

## なぜ Sign を作るのか

Sign は、一言で言うと

> **「AIがベアメタルを書くための言語」**

である。

---

## 1. ターゲットユーザー

Sign のプライマリユーザーは**AI（大規模言語モデルおよびコード生成システム）**である。
セカンダリユーザーとして、その出力をレビューする人間エンジニアを想定している。

人間は Sign を「ぎりぎり読める」。しかし「書こう」と自発的になる人間は少ない。
これは欠点ではなく、**設計上の意図**である。

AIは：
- 数学的・圏論的な構造を自然に扱える
- 例外のない決定論的なセマンティクスを好む
- 構文の曖昧さに弱い → 最小構文が理想
- 型の正しさをコンパイル時に検証してほしい
- OSなしの環境でも動くコードを生成したい

Sign はこれらすべてに応える。

---

## 2. なぜベアメタルか

Sign は Cのエコシステムに依存しない独自のエコシステムを持つ。
これにより、**リンカスクリプトの制約から解放**され、言語自身が：

- BIOS / UEFI の起動シーケンスを記述できる
- FPU・SIMD の初期化状態を型として管理できる
- RAMの大半をスタックに割り当てるような設計が可能

「AIが書いたコードが、OSのない環境で動く」という状況が実現できる。
これは AI が自律的なハードウェア制御コードを生成する未来に直結する。

---

## 3. 設計思想と AI 親和性の対応

| Sign の設計 | AI にとっての利点 |
|-------------|-----------------|
| **予約語ゼロ** | 「これは予約語か？」という判断が不要。生成コードの衝突リスクがない |
| **Unit（`__`）伝播** | 例外なし。パターンが数学的に決定論的 → 生成が安全 |
| **型が構文から自明（O(n)）** | ASTを生成するだけで型が確定。制約ソルビング不要 |
| **layer制約** | `layer < 2` で float を書けば確実にコンパイルエラー。ハードウェア的誤りを静的に防止 |
| **`'` チェーン** | 多次元アクセスが数学的な写像の合成として表現できる |
| **alloca ファースト** | ヒープなしでクロージャ・カリー化が実現できる。GC 不要 |
| **`Point@` 型注釈** | 型推論が「型名を見るだけ」で完結。LLM が型を明示しやすい |

---

## 4. 人間との協働

AI が Sign を書き、人間がレビューする、という分業が想定されている。

- **AI** → Sign コードを生成（数学的正確さを担保）
- **人間** → 設計意図・アーキテクチャを定義、出力をレビュー
- **コンパイラ** → layer 制約・型制約でハードウェア的誤りを検出

この三者の協働により、**ベアメタルから OS まで、安全に AI が記述するスタック**が成立する。

---

## 5. 「ぎりぎり読める」の意味

Sign の構文は極めてミニマルであり、制御構文（`if`, `for`, `while`）が存在しない。
代わりに `__`（Unit）の伝播と `&`/`|` の短絡評価で条件分岐を表現する。

これは人間にとって学習コストが高い。しかし：

- **読むこと**は、慣れれば数学式と同じ感覚でできる
- **書くこと**は、数学的思考が自然にできる人（or AI）に向いている

「読める人間が、書く AI を監督する」という構造は、
現代の AI 活用の文脈でも自然なモデルである。

---

## 6. まとめ

Sign は：

1. **AI がコード生成するための言語**（プライマリ）
2. **ベアメタルから OS まで書ける**（layer: 0 〜 4）
3. **Cのエコシステムに依存しない独自スタック**
4. **数学的・圏論的な基盤による形式的な正しさ**
5. **人間がレビュー可能なミニマル構文**
6. **命令と言語の写像が仕様として定義されている**

という6つの柱で設計されている。

---

## 7. 命令と言語の写像関係を設計し直す

### C エンジニアからの正当な問い

> 「alloca のリンクドリスト、ポインタ追跡のコストは意味があるのか？」

これは正しい問いである。愚直な実装が最適化済みの C より遅いなら、Sign を使う理由がない。

### C の暗黙の前提

C は「コンパイラと人間が協力して最適化する」という前提を持つ：

```c
// C: 「最適化されることを期待するコード」
// → 何が生成されるかは実装依存
int result = a + b * c;
```

言語構文と機械命令の対応は **暗黙・不定** であり、
「最適化される可能性があるコード」を書くのが C のスタイルである。

### Sign の設計原則

Sign は「言語構文と命令が定義された対応を持つ」という原則を取る：

| Sign の構文・機能 | 対応する命令パターン |
|------------------|-------------------|
| Unit 伝播（`__`） | `AND`/`OR` の短絡評価。条件分岐命令（`JE`/`JNE`）なし |
| `'` チェーン | ポインタ加算のみ（ゼロコピー） |
| `layer` 制約 | 使用可能な命令セットを型レベルで限定 |
| alloca 連続ブロック | `sub rsp, N` の一命令でスタック確保 |
| レンジ式（`layer: 3`） | SIMD gather/scatter 命令に変換 |

### 「コストが言語仕様として定義される」意味

Cは「何を書いたら何のコストがかかるか」が文脈・最適化レベル・コンパイラに依存する。

Sign は**言語仕様の中に命令コストの対応が含まれる**。
これにより：

- AI がコストを計算しながらコードを生成できる
- 人間がレビュー時にコストを予測できる
- 「最適化のブラックボックス」に依存しない

これはゼロコスト抽象化の更に先にある考え方：
**「抽象化のコストがゼロ」ではなく「抽象化の命令コストが仕様として自明」。**
