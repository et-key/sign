# Sign システムアーキテクチャ仕様：レイヤー意味論と随伴によるメモリ・リンクの完全統合

## 1. イントロダクション：第一原理

Signは、高水準の宣言的プログラムからOSカーネルなどの低水準ハードウェア制御までを同一の文法と一貫した数理モデルで記述する。
この一貫性を支えるのが、**「レイヤーによる意味論のシフト」** と **「エクスポート（前置）とインポート（後置）の随伴関係」** である。

従来の言語（CやRustなど）では、メモリ管理、モジュール解決、リンク戦略、ハードウェアI/O（volatile等）が個別の言語機能や外部ツール（リンカ、ビルドシステム）に分断されていた。
Signでは、これらすべてを型システムと演算子の随伴関係に統合し、ビルド設定ファイル（`option.ms`）による最小限の環境宣言のみで解決する。

---

## 2. 随伴関係によるメモリ配置とリンクの自動導出

Signの公開（エクスポート）と要求（デマンド/インポート）は、圏論的な**随伴（Adjunction）**の関係にある。

```
[エクスポート] (前置/定義側)   ⊣   [デマンド] (後置/参照側)
      # / ## / ###                      @
```

### 2.1 随伴ペアとリンク・メモリの対応

定義名に前置される `#` 演算子と、参照名に後置される `@` 演算子は、設定された `layer` および `link` 戦略に基づき、コンパイラによって以下のように自動的にリンク手段と物理・仮想メモリ配置へとマッピングされる。

| エクスポート側 | デマンド側 | 物理/仮想配置 | リンク解決方式 |
| :--- | :--- | :--- | :--- |
| **`#foo : expr`** (Rc) | **`foo@`** | project-arena（ローカルバンプ/スタック） | 静的に解決（インライン化または同一オブジェクト内） |
| **`##foo : expr`** (Arc) | **`foo@`** | shared-heap（共有仮想メモリ） | `link: static` なら静的結合、`link: dynamic` なら動的共有 |
| **`###foo : expr`** (Pin) | **`foo@`** | pinned-area（固定仮想メモリ / 物理アドレス固定） | FFI境界、物理空間、または常駐セクションへの静的/動的結合 |

### 2.2 リンカスクリプトの代替性

この随伴関係により、どのシンボルがどのメモリ性質（ローカル、共有、固定）を持つかをコンパイラが完全に静的把握できる。
そのため、C言語等で必要だった複雑なリンカスクリプト（`.text`、`.data`、`.bss` のセクション配置指示）はOption.msファイルに代替される。コンパイラは随伴関係から各オブジェクトのセクション分類を自動導出し、`option.ms` で指定された `base`（テキスト開始アドレス）を起点にメモリマップを自己構築する。

## 3. レイヤー（Layer）による意味論のシフト

Signでは、ビルドオプションの `layer`（`0`: bare, `1`: alloc, `2`: fpu, `3`: simd, `4`: std）の宣言により、同一の演算子がハードウェア物理層から高水準OS層の意味論へとシームレスにシフトする。

### 3.1 `@` (Load/Demand) と `#` (Store/Allocation) の意味論シフト

#### layer 0 (bare) ：物理・ハードウェア制御層
- **前置 `@`（`@ ptr`）/ 中置 `#`（`ptr # val`）**：暗黙的に **volatile read / volatile write** にシフトする。ハードウェアMMIOレジスタへの最適化による消失を防ぐため、常に `ptr::read_volatile` / `ptr::write_volatile` 相当としてコンパイルされる。
- **前置 `#`（`# expr`）/ `##`**：コンパイルエラー。ヒープが存在しないため、Rc/Arcの生成は禁止される。
- **後置 `@`（`foo@`）**：静的に解決可能なシンボルの要求（静的リンク）のみ有効。

#### layer 1 (alloc) / layer 2 (fpu) / layer 3 (simd) ：仮想メモリ・ローカルアロケータ層
- **前置 `@` / 中置 `#`**：通常のメモリロード/ストア（`*ptr` / `*ptr = val`）として扱われる。
- **前置 `#` / `##`**：有効。それぞれ `Rc<RefCell<T>>` および `Arc<Mutex<T>>` 相当のメモリ確保として機能する。
- **後置 `@`**：静的リンク解決のみ有効。OSローダーが存在しないため、動的共有リンクはコンパイルエラーとなる。

#### layer 4 (std) ：OS・ダイナミックランタイム層
- **前置 `@` / 中置 `#`**：通常のメモリロード/ストア。
- **前置 `#` / `##`**：有効。
- **後置 `@`**：動的モジュール解決（`dlopen` 相当の実行時デマンド）が有効化される。`link: dynamic` 設定時はOSのダイナミックローダーにシンボル解決を委ねる。

### 3.2 レイヤー意味論の遷移マトリクス

| 演算子の用法 | layer 0 (bare) | layer 1〜3 (alloc/fpu/simd) | layer 4 (std) |
| :--- | :--- | :--- | :--- |
| **中置 `#`** (`ptr # val`) | `write_volatile` | `*ptr = val` (通常書込) | `*ptr = val` (通常書込) |
| **前置 `@`** (`@ ptr`) | `read_volatile` | `*ptr` (通常読込) | `*ptr` (通常読込) |
| **前置 `#` / `##`** (alloc) | ❌ コンパイルエラー | ✅ `Rc` / `Arc` 確保 | ✅ `Rc` / `Arc` 確保 |
| **後置 `@` (link: static)** | ✅ 静的シンボル解決 | ✅ 静的シンボル解決 | ✅ 静的シンボル解決 |
| **後置 `@` (link: dynamic)**| ❌ コンパイルエラー | ❌ コンパイルエラー | ✅ 動的共有シンボル解決 |

## 4. ビルドシステム（`option.ms`）との統合

> [!IMPORTANT]
> **`option.ms` の全フィールド仕様は [`Option_MS_Spec_ja-jp.md`](Option_MS_Spec_ja-jp.md) を参照すること。**
> 本ドキュメントは `#`/`@` 演算子の layer 別意味論と随伴関係によるリンク自動導出の数学的基盤を記述する。

システムレイヤーを記述する際、これらの振る舞いはすべて `option.ms` のシンプルな定義に集約される。

### 4.1 静的配置におけるメモリ境界 (link: static)
組み込みやOS起動初期段階のように物理メモリ境界を手動で決定する場合、`link: static` サブブロックを用いて ROM/RAM のアドレス範囲を宣言する。

```son
` stm32等の組み込みターゲットでの宣言例
layer : 0
link :
	static :
		memory :
			rom : origin 0x08000000  length 1024K
			ram : origin 0x20000000  length 128K
```

### 4.2 実行時マップ取得 (auto)
PCやSBCのように、実行時（UEFIやデバイスツリー経由）にメモリサイズが決定される場合は、`ram: auto` と記述し、領域の決定をターゲットごとのスタートアップランタイムに委ねる。

```son
` OSカーネル開発での宣言例
 layer : 0
link :
	static :
		memory :
			rom : origin 0x00000000  length 4M
			ram : auto   ` UEFI等のシステムコール/パラメータから起動時に自動マッピング
```

### 4.3 ヒープ領域の一括確保 (heap: max)

`layer: 1` 以上で動的メモリ管理を行う場合、
`heap: max` を宣言することで起動時に指定サイズを一括確保し、
内部はバンプポインタアリーナとして管理する。

```son
` layer: 1 以上でのヒープ一括確保
 layer : 1
link :
	static :
		memory :
			ram  : origin 0x20000000  length 128K
			heap : max 64K   ` 起動時に 64K を一括確保、内部はアリーナで管理
```

**動作：**
- コンパイラが `boot.sn` に起動時の一括確保コードを自動生成
- 確保済みブロック内でバンプポインタで管理（O(1) アロケーション）
- `free` 不要（スコープ終了時にアリーナ全体をリセット）
- 連続メモリのためキャッシュ効率が高い

```sign
` boot.sn に自動生成されるヒープ初期化コード（概念）
`"layer:1"
"
	` heap: max 64K の定義から
	ヒープ領域を alloca で一括確保
	heap_base : alloca 65536   ` 64K
	heap_ptr  : heap_base      ` バンプポインタ
"
```

| 確保方式 | アロケーションコスト | フラグメント | free 操作 |
|------------|---------------|------------|----------|
| 都度 `malloc` | O(1)～展選 | 発生する | 必要 |
| `heap: max` 一括確保 | **O(1) バンプ** | **発生しない** | **不要** |

## 5. 設計の意義：OSが書ける高水準関数型言語へ

Signは、数学的に純粋な「零対象（Unit）の伝播」と「関数の随伴」をベースにしながら、コンパイラオプション（レイヤーとリンク）の切り替えだけで、ベアメタルハードウェアの volatile 制御から、OS上の動的ライブラリロードまでを矛盾なくカバーする。

この設計により：
1. **不要な言語機能の排除**：`volatile` 専用のキーワードや演算子を必要とせず、レイヤーコンテキストだけで `@` と `#` の物理的意味を切り替える。
2. **安全性の段階的保証**：OS開発者は、起動ルーティンを `layer: 0` で厳密に（ヒープなし・volatile自動で）書き、メモリ管理が立ち上がった後は `layer: 1` へとシームレスに移行できる。
3. **完全な自己完結性**：外部のリンカスクリプトやC言語のヘッダファイルモデルを完全に排除し、`option.ms` と `.sn` ソースコードだけでシステム記述を完結させる。
