# コプロダクト・リゾルバー（余積解決器）仕様

本仕様書は、構文解析直後の段階において、スペース ` ` で区切られたフラットな並び（`coproduct_block`）を型タグの優先順位に基づいて二分木ASTへと貪欲に縮約（Reduction）するアルゴリズムについて規定します。

---

## 1. 概要とデッドロックの回避

Sign言語において、スペースによる並置は「余積（Coproduct）」を意味し、型に応じて`reverse_apply`（逆適用）、`apply`（関数適用）、`compose`（関数合成）、`concat`（リスト連接/双積）のいずれかに還元されます。

型推論（Algorithm W）の完了を待ってからASTを構築しようとすると循環依存が生じるため、本リゾルバーは**構文解析直後に各ノードに「やんわり」と付与された型タグ（構造的・構文的種類情報）**を用いて、決定論的に結合関係を解決します。

---

## 2. 項の型カテゴリ判定（getCategory）

解析対象の各ノード `node` に対し、型推論前のおおまかなカテゴリ（`Lambda` または `Atom`）を以下のルールに従って判定します。

- **`Lambda`（関数・射）と判定される条件**:
  - ノード自身に `isLambda` フラグが立っている。
  - `?`（関数定義）や `compose`（合成ノード）、部分適用状態の `operation`（オペランドが一部欠けている中置・前置・後置演算子など）。
  - `print`, `<print>`, `_` 等の定義済みの組み込み関数名。
  - 前置演算子 `@` が適用されている項。
- **`Atom`（値・対象）と判定される条件**:
  - 数値リテラル、文字列リテラル。
  - 環境マップ `env` に登録された識別子のうち、カテゴリが `Atom` であるもの。
  - 算術演算やリスト連接 `concat` 等の通常の演算ノード。
  - その他上記に該当しないもの。

---

## 3. 優先順位（Precedence）の定義

隣接するペア `(LHS, RHS)` の型カテゴリ組み合わせに対して、以下の結合優先順位を定義します（優先度の高い順に処理されます）。

| 優先度（降順） | 左辺型 (`leftCat`) | 右辺型 (`rightCat`) | 還元先ノード | 意味 |
| :--- | :--- | :--- | :--- | :--- |
| **10.3** | `Atom` | `Atom` | `concat` | 直和/双積（リストの連接） |
| **10.2** | `Lambda` | `Lambda` | `compose` | 射の合成（関数合成） |
| **10.1** | `Lambda` | `Atom` | `apply` | 射の適用（関数適用） |
| **10.0** | `Atom` | `Lambda` | `apply_reverse` | 逆適用（右辺の関数に左辺の値を適用） |

---

## 4. 総当たり優先順位解決アルゴリズム（Reduction）

フラットな項の並び `items`（例: `a b c d` に対する型タグ `Ta Ub Vc Wd`）に対して、以下の手順で二分木を構築します。

1. **優先順位テーブルの走査**:
   優先度定義リスト（10.3 から 10.0 まで）を上から順に処理します。
2. **リストの左結合スキャン**:
   現在適用中の優先度 `prec` に対し、`items` の左から順に隣接するペア `(items[i], items[i+1])` を走査します。
3. **マッチングと縮約 (Reduction)**:
   - 各ペアの型カテゴリ `(catL, catR)` が `prec` の条件と一致した場合、その2つの項を取り出して新しい `operation` ノード（`concat`, `compose`, `apply`, または `apply_reverse`）に結合します。
   - 例: `Ta Ub` の優先順位が最も低く（10.0）、隣接する `Ub Vc` の優先順位がより高い（10.3等）場合、先に `Ub Vc` の結合が実行されて `a (b c) d` に縮約されます。
   - 縮約が実行された場合、リストのインデックスを更新して、再びリストの最初から再スキャンを行います（左結合性の保証のため）。
4. **収束**:
   すべての優先度レベルの処理が終了したとき、リストは単一のルートノードに収束します。

この総当たり優先度ディパッチにより、曖昧なスペース結合関係が型タグに基づいて一意かつ健全に解決されます。

## 5. リスト・構造体（Dict）の暗黙マージ回避と後置チルダ（~）による静的型保証

### 5.1 チルダ必須化について
`List` 同士や `Struct`（ブラケットで囲まれた辞書・オブジェクトなど）同士が暗黙的にマージされると、静的な型推論時に型の境界が曖昧になり、動的型付けを導入せざるを得なくなります。
これを防ぎ、静的型付け（コンパイル時型保証）を確実にするため、**「`List` 同士または `Struct` 同士の連接（concat）においては、双方に後置チルダ `~`（展開/評価強制）が付与されている場合のみ結合を許可する」**というルールを適用します。

### 5.2 結合の条件と評価のセマンティクス

1. **`struct~ struct~` または `list~ list~` (双方に後置チルダあり)**
   - 両辺にチルダがあるため、`concat` 結合（マージ）が実行され、単一のフラットな構造体またはリストに結合されます。
2. **`struct struct` または `list list` (双方に後置チルダなし)**
   - `concat` 結合は実行されず、別個の `Atom` として解決されます。
   - これは低レイヤ（WebAssembly等）から見ると**「2つのポインタ（参照）のリスト」**と等価であり、コンパイル時にそれぞれの型が完全に保持され、安全に型が保証されます。
3. **`struct~ list~` / `list~ struct~`**
   - 多層型とみなされ、`struct`に結合されます。

### 5.3 リゾルバーでの実装処理

コプロダクト・リゾルバー内の `concat` 還元処理において、両辺の型が `List` / `Struct`（`bracket` ブロックや `coproduct_block` 等を含む）であると判定された場合、次のチェックを行います。

```javascript
// coproduct_resolver.js の実装ロジック
if (isListL && isListR) {
  const leftHasTilde = (left && left.type === 'operation' && left.operator === '~' && left.position === 'postfix');
  const rightHasTilde = (right && right.type === 'operation' && right.operator === '~' && right.position === 'postfix');
  if (leftHasTilde && rightHasTilde) {
    // 双方にチルダがある場合のみ、連接（concat）として結合を許可する
    newNode = { type: 'operation', operator: ' ', left, right, name: 'concat' };
    items.splice(i, 2, newNode);
  } else {
    // チルダがない、または片方にしかない場合はマージせず、この時点での結合をスキップする
    continue;
  }
}
```

この仕様追加により、開発者は意図しない暗黙のリストマージを回避しつつ、静的型解決とWebAssemblyでのゼロコストな参照渡し（ポインタ渡し）の恩恵を受けることができます。
