# Sign 実行モデル：すべては main の内部関数である

## 核心

> **Sign のすべての関数は、`main.sn` という1つの関数の内部関数として静的に展開される。**

これは Sign の設計における最も根本的な性質であり、以下のあらゆる設計判断の根拠となる。

---

## 1. ファイル名 = エントリポイント = 関数名

Sign において、`.sn` ファイルは関数定義である。ファイル名がそのまま関数名になる。

```
add.sn       →  関数 add の定義
multiply.sn  →  関数 multiply の定義
main.sn      →  プログラムのエントリポイント（= C の main()）
```

`main.sn` はプログラム全体の「最外スコープ」であり、
他のすべての `.sn` ファイルは `main` の内部関数として取り込まれる。

---

## 2. インポートは「内部関数の定義」である

```sign
`add.sn`@~   ← add.sn を「インポート」する
```

これは「ファイルを読む」のではなく、**`add` という内部関数を現在のスコープに静的に定義する**操作である。

C で言えば：

```c
int main() {
    // add.sn のインポート = add を内部関数として定義
    int add(int a, int b) { return a + b; }  // C の nested function（GNU 拡張）

    // main.sn の本体
    int result = add(3, 5);  // → 8
    return 0;
}
```

---

## 3. 部分適用は compile-time の静的特殊化である

```sign
add3 : add 3
add3 5   ` → 8
```

`add3` は「ランタイムにクロージャを生成する」のではなく、
コンパイラが **`add` を `3` で特殊化した関数 `add3`** を `main` 内に静的に生成する。

```c
// コンパイラが生成するコード（概念）
int main() {
    int add(int a, int b) { return a + b; }

    // add3 : add 3  → コンパイル時に特殊化
    int add3(int b) { return add(3, b); }  // ← 静的に展開

    int result = add3(5);  // → 8
}
```

実行時のクロージャ構造体は存在しない。ヒープ確保も alloca も不要。

---

## 4. クロージャのエスケープ問題が存在しない理由

「クロージャがスタックフレームを超えて生きる」問題（dangling pointer）は、
クロージャを**ランタイムに動的生成**する言語のものである。

Sign では：
- すべての「クロージャ」はコンパイル時に特殊化された内部関数
- 内部関数の寿命 = `main` の寿命 = プログラム全体
- スタックフレームを超えるという概念が設計上存在しない

---

## 5. 帰結：完全なヒープレス

この実行モデルにより、**コア言語（layer 0）はヒープを一切必要としない**：

| 機能 | 確保方法 | 理由 |
|------|---------|------|
| 完全適用の関数呼び出し | レジスタ渡し | 静的に確定 |
| 部分適用 | コンパイル時特殊化 | 実行時クロージャ不要 |
| 配列・構造体 | `alloca` 連続ブロック | スコープ終了で自動回収 |
| モジュールスコープの変数 | `main` のスタックフレーム | プログラム全体が寿命 |

GC も `free` も所有権注釈も不要。すべてがコンパイル時に決定する。

ただし、`memory_management.md` が定義する stdlib 拡張（Shared-Heap, `##`）は任意選択の layer 1 以上の機能であり、この原則の例外として明示的にオプトインされる。RC/ARC 管理された Shared-Heap は、コア言語のヒープレス性を損なわない、独立した上位レイヤーの機能である。

---

## 6. layer との関係

この実行モデルは `layer: 0`（bare metal）から成立する。
OSなし・ランタイムなしでも、`main` 関数とその内部関数群が完全に動作する。

```
layer: 0  → main + 内部関数群が alloca とレジスタ渡しで動く
layer: 1  → RAM が使えるようになる（alloca の連続ブロックがより活用可能に）
layer: 2+ → FPU, SIMD 等が加わる
```

Sign のスタックが「BIOS から書ける」のは、この実行モデルが OS のランタイムに
依存しないことの直接の帰結である。
