# Sign の末尾呼び出し最適化（TCO）仕様

## 概要

Sign にはループ構文（`for`/`while`）が存在しない。
反復処理は**再帰**と**レンジ式**で表現する。

したがって、再帰がスタックオーバーフローを起こさないことは**言語の基本保証**である。
コンパイラは末尾位置にある呼び出しを `CALL` ではなく `JMP` に変換する（**TCE: Tail Call Elimination**）。

---

## 1. 末尾位置の定義

関数 `f` の本体で、「その呼び出しの結果がそのまま関数の返値になる」位置を**末尾位置**と呼ぶ。

Sign では `?` 演算子の右辺（本体）全体が評価対象であり、
`&`/`|` の各節の**最後に評価される式**が末尾位置になる。

### 末尾位置の例

```sign
` f の呼び出しが末尾位置にある
loop : n acc ?
    n == 0 & acc |          ` ← acc が末尾（再帰でない → TCO不要）
    loop (n - 1) (acc + n)  ` ← loop が末尾位置 → TCO対象
```

### 末尾位置でない例

```sign
` 呼び出しの結果をさらに演算に使う → 末尾位置でない
bad : n ?
    (f n) + 1   ` ← f n の結果に +1 している → f n は末尾位置でない
```

---

## 2. `&`/`|` チェーンと末尾位置

Sign の条件式 `cond & then | else` は以下のように評価される：

```
cond が Unit でない → then を評価して返す
cond が Unit       → else を評価して返す
```

`then` と `else` の**両方が末尾位置**である。したがって、どちらの節にある再帰呼び出しも TCO 対象になる。

```sign
` &/| の各節が末尾位置
classify : n ?
    n < 0  & neg  |    ` neg が末尾（リテラル、TCO不要）
    n == 0 & zero |    ` zero が末尾（リテラル、TCO不要）
    pos              ` pos が末尾（リテラル、TCO不要）
```

```sign
` 再帰呼び出しが各節の末尾にある
collatz : n steps ?
    n == 1     & steps            |  ` 基底ケース
    n % 2 == 0 & collatz (n / 2)  (steps + 1) |  ` 偶数 → TCO
                 collatz (n * 3 + 1) (steps + 1)   ` 奇数 → TCO
```

各 `|` 節の末尾呼び出しは**すべて独立に TCO 対象**となる。

---

## 3. 相互再帰の TCO

2つ以上の関数が互いに呼び合う**相互再帰**も、呼び出しが末尾位置にあれば TCO 対象。

```sign
` 相互再帰（どちらの呼び出しも末尾位置）
is_even : n ? n == 0 & 1  | is_odd  (n - 1)
is_odd  : n ? n == 0 & __ | is_even (n - 1)
```

```
is_even(1000000):
  → JMP is_odd(999999)     ← CALL ではなく JMP
  → JMP is_even(999998)
  → JMP is_odd(999997)
  → ... （スタック蓄積なし）
  → is_even(0) → 1
```

スタックフレームは常に1枚だけ。スタックオーバーフローは発生しない。

---

## 4. コンパイラの変換規則

### 単純末尾再帰

```sign
f : args ? cond & base | f new_args
```

コンパイル後（概念）：

```c
// CALL を JMP に変換
f:
    if (!cond) goto f_with_new_args;  // JMP（スタックフレーム再利用）
    return base;
```

### 複数分岐の末尾再帰

```sign
f : n ?
    cond1 & f a1 |
    cond2 & f a2 |
            f a3
```

各分岐がそれぞれ独立に JMP に変換される。

---

## 5. TCO とレンジ式の関係

反復処理の**推奨表現はレンジ式**であり、TCO に依存しない：

```sign
` レンジ式（TCO 不要）
sum : [1 ~ n] $[acc x ? acc + x] 0

` 再帰（TCO が必要）
sum_rec : n acc ? n == 0 & acc | sum_rec (n - 1) (acc + n)
```

レンジ式はコンパイラが直接ループ命令（`LOOP`/`JNZ` 等）に変換するため、
スタックを使わない。再帰より効率的な場合がある。

**ガイドライン**：
- 配列・シーケンスの処理 → レンジ式を優先
- 状態機械・木構造の走査・相互再帰 → 再帰 + TCO

---

## 6. TCO の静的保証

Sign のコンパイラは、末尾位置の再帰呼び出しに対して**必ずTCOを適用する**。
これは最適化ではなく**言語仕様としての保証**である。

- ユーザーが `@tailrec` 等のアノテーションを書く必要はない
- 末尾位置にない再帰はコンパイラが警告を出す（深いスタックが予測される場合）
- 静的アリティと実行モデル（全関数が `main` の内部関数）により、
  TCO の対象を静的に決定できる

---

## 7. 実行モデルとの整合

「全関数が `main` の内部関数」であるため、
関数 `f` から `f`（または他の内部関数）への末尾呼び出しは、
コンパイラが**同一スタックフレームへの JMP** として扱える。

これは C の setjmp/longjmp や Scheme の continuation とは異なり、
スタックを明示的に操作する機構を必要としない。
コンパイラの命令選択（`CALL` → `JMP`）だけで実現できる。
