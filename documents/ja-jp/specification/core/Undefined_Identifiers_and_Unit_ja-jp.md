# Sign言語における未定義識別子とUnitの挙動に関する仕様

## 1. 未定義識別子のUnit収束（予約語の完全排除）

Sign言語には `null`, `nil`, `false` などの予約語は存在しません。
**「現在スコープにおいて定義されていない識別子はすべて `__`（無）として評価される」**というコア仕様が存在します。

### 1.1 仮想キーワードとしての活用
この仕様により、ユーザーは任意の英単語やUnicode記号をゼロコストの「仮想キーワード」として利用できます。
```sign
@lazy tick
@iterator next
```
上記のように、未定義の識別子（`tick`, `next`）を関数呼び出しに渡した場合、それらは安全に `__` として評価されるため、可読性を高めるための自然言語的なマーカーとして機能します。

## 2. UnitとFalseの同型性 (Isomorphism) と短絡評価

圏論的論理値の設計仕様にある通り、Signでは「無（Unit）および空リスト」のみが偽（False）として振る舞い、それ以外（0を含む）は真（True）として機能します。

### 2.1 論理演算子による自然な制御フロー
言語組み込みの論理積（`&`）および論理和（`|`）は短絡評価（Short-circuit）を行います。
これにより、従来の `if` 文や `match_case` などの制御構文を用いずに、純粋な数式として条件分岐や再帰の終了条件を定義できます。

```sign
` 例: appendの終了条件と再帰ステップ
` リスト（xs）自身が論理値として機能するため、空チェックに == や != の比較演算子は不要です
append : [x ~xs] [y ~ys] ?
    xs & x (append xs (y ys)) | @x y ys
```
- `xs` が空リストではない（真）場合、後続の再帰ステップ `x (append xs (y ys))` が評価されます。
- `xs` が空リスト（`__`／偽）の場合、左辺全体が偽となり、短絡評価のフォールバックとして `|` の右辺（終了条件 `@x y ys`）が評価されます。

## 3. 無効化された演算子と Unit の役割

### 3.1 演算子の安全な無効化
特定のスコープにおいて演算子（例: `+`）の使用を禁止し、かつパニックを起こさずに処理を遮断したい場合、演算子を以下のようにオーバーライドします。

```sign
[+] : ~x ? __
```
可変長引数 `~x` によってオペランドをすべて吸収し、明示的に `__` を返すことで、演算子が用いられた式全体を安全に `__` に崩壊（ショートサーキット）させます。
※ `[+] : __` と定義した場合、`__` が恒等射（id射）として機能し、オペランド同士が意図せず結合（関数適用）されてしまうため、必ず `~x` を用いて引数を吸収する必要があります。

## 4. `__` (Unit) と `$__` (Unitのアドレス) の挙動の違い

評価の中断や遅延において、この2つは明確に異なる役割を持ちます。

### 4.1 `__` (即時崩壊・墓標)
- **性質**: その計算パスの「死（完全な終了）」を表します。
- **挙動**: Unit伝播によって即座に式全体を飲み込み、ショートサーキットを引き起こして計算を消滅させます。

### 4.2 `$__` (遅延サスペンド・終端シグナルのPromise)
- **性質**: 計算の「休眠（サスペンド）」を表すPromise/Thunkです。
- **挙動**: 消費側が明示的に `@` で要求（デリファレンス）するまで安全に待機します。`fold_lazy` のような遅延評価ストリームにおいて、例外を投げることなく「終端（End of Stream）」を美しく伝達するためのシグナルとして用いられます。

### 4.3 `$__` の数学的同型性：統一説明

> [!NOTE]
> **`$__` は「Unit のアドレスを取ると Unit になる」と「ストリームの終端」が数学的に同型であることから、同一記号で両方を記述できる。**
>
> - **基本定義**：`$__` = Unit のアドレス = `__`（零対象の性質から、自分自身への唯一の射が存在する）
> - **ストリーム終端**：`@` で要求すると `__` を返す「待機状態」→ 要求後の結果が `__` = Unit と同型
> - **同型の意義**：「何もない（そこにアドレスするものがない）」と「終わり（それ以上先に進めない）」は結果として同じ `__` を得る。
> これは言語実装における数学的同型が言語設計の一貫性を自然に保証したものである。
