# Sign 値表現仕様：文字 boxing と Unit の物理表現

## 1. 概要

Sign 言語の文字型（Char）は **Modified UTF-8（MUTF-8）** をエンコーディング標準として採用する。
MUTF-8 は Java VM・Android Dalvik・Kotlin JVM 等で実績のある規格であり、
U+0000（ヌル文字）を `0xC0 0x80`（2バイト）として符号化することで
**生の `0x00` バイトがストリームに現れない** という重要な性質を持つ。

> [!IMPORTANT]
> **Sign における文字の「boxing」とは MUTF-8 の多バイトエンコーディングそのものを指す。**
> 型情報は先頭バイトのビットパターン（`0xxxxxxx` / `110xxxxx` / `1110xxxx`）に自然に埋め込まれる。
> 既存の Unicode 標準の上に自然に実現された Boxing の作法である。

---

## 2. MUTF-8 エンコーディング規則

| コードポイント範囲 | バイト列 | 先頭バイト |
|:---:|:---:|:---:|
| U+0001 〜 U+007F | `0xxxxxxx` | 1バイト（ASCII 互換） |
| **U+0000** | `11000000 10000000`（`0xC0 0x80`） | **2バイト（MUTF-8 固有）** |
| U+0080 〜 U+07FF | `110xxxxx 10xxxxxx` | 2バイト |
| U+0800 〜 U+FFFF | `1110xxxx 10xxxxxx 10xxxxxx` | 3バイト |
| U+10000〜U+10FFFF | サロゲートペア経由（WTF-8 準拠） | 各3バイト × 2 |

> [!NOTE]
> MUTF-8 において先頭バイトの上位ビットが「型タグ」として機能する。

---

## 3. `0u00`（ヌル文字 U+0000）の boxing

```
MUTF-8 表現：
  byte[0] = 0xC0  （11000000）→ 2バイトシーケンスの開始
  byte[1] = 0x80  （10000000）→ 継続バイト、payload = 000000 = U+0000

生の 0x00 バイトは一切含まれない。
```

### 3.1 `$0u00` の指すもの

`$0u00` は Sign コンパイラが rodata 内に配置した **MUTF-8 エンコード済みの `0u00` シーケンス** のアドレスを返す。

```
Sign rodata（コンパイラ管理）：
┌──────┬──────┐ ← $0u00 が指すアドレス
│ 0xC0 │ 0x80 │  （2バイトの MUTF-8 boxing）
└──────┴──────┘
```

- 物理番地 0x00 とは無関係
- ARM のリセットベクタ・stdin（fd=0）と衝突しない
- C 文字列終端（NUL バイト）とも衝突しない

---

## 4. Unit（`__`）と `0u00` の関係

### 4.1 同型性と非同一性

```
__ ≅ 0u00  （圏論的に同型：どちらも零元）
__  ≠ 0u00  （物理表現・boxing バイト列は異なる）
```

| 性質 | `__`（Unit） | `0u00`（ヌル文字） |
|:---:|:---:|:---:|
| MUTF-8 表現 | 存在しない（Unit は文字でない） | `0xC0 0x80` |
| `$` で得るもの | Unit 専用 rodata スロットのアドレス | `0xC0 0x80` シーケンスのアドレス |
| 零元としての役割 | 全型の零対象 | Char 型の零元 |
| 双代数の単位元 | ✓（全域） | ✓（Char 型文脈のみ） |

### 4.2 `$__ = $0u00` の意味

`$__ = $0u00` は実装上の同一アドレスを意味するのではなく、
**概念的・代数的な同型性**の表明である。

どちらも「その型の零元のアドレス」であり、Sign コンパイラはこの同型性を利用して
Unit と Char の相互変換（型キャスト）を定義できる。

---

## 5. Unit の物理表現と `$__`

Unit は文字型ではないため、MUTF-8 エンコーディングを持たない（Value が無い）。
Sign コンパイラは **Unit 専用の静的スロット** を rodata 内に確保する。

```
Sign rodata（コンパイラ管理）：
┌─────────────────────────┐ ← $__ が指すアドレス
│  [Unit マーカー]        │  （実装依存・コンパイラが定義する Unit 識別値）
└─────────────────────────┘
```

- 書き込み不可（read-only）
- `$__ # X` → この領域への書き込み → 致命的エラー
- `@__` → Unit マーカーを読み出す → `__` に収束（Layer 依存の安全性あり）

---

## 6. Id射（`!__`）の dispatch

### 6.1 Id射は boxing に格納しない

`0xC0 0x80` の 2 バイトは MUTF-8 のエンコーディングであり、関数ポインタを含まない。
Id射の振る舞いは **コンパイラが静的に生成する**（ゼロコスト）。

```sign
` __ | X のコンパイル方式（概念）：
`   左辺が Unit → 右辺をそのまま返す（Id射の等価操作）
`   左辺が非Unit → 左辺をそのまま返す（短絡評価）
```

### 6.2 `!__` の実体

`!__` は「Unit を受け取り引数をそのまま返す純粋関数」としてコンパイラが生成する関数オブジェクト。
型は `∀A. A → A`。`__`（零対象）とは異なる非 Unit 値（真）である。

---

## 7. I/O との衝突まとめ

| 値 | MUTF-8 表現 | 生の 0x00 | ARM ベクタ衝突 | stdin(fd=0)衝突 |
|:---:|:---:|:---:|:---:|:---:|
| `0u00` | `0xC0 0x80` | なし | なし | なし |
| `__` | （存在しない） | なし | なし | なし |
| `"hello"` | UTF-8 準拠 | なし（NUL 不使用） | なし | なし |

MUTF-8 採用により、Sign のすべての文字・文字列は I/O を通過しても安全。
