# Sign 値表現仕様:文字 boxing と Unit の物理表現

## 1. 概要

Sign 言語の文字型(Char)は **標準 UTF-8(RFC 3629 準拠)** をエンコーディングとして採用する。
MUTF-8 のような独自拡張・overlong 表現は一切用いない。UTF-8 が overlong 表現を禁止しているのは、
規格上の些末な制約ではなく、任意の 1 バイトからその役割(単独文字/継続バイト/先頭バイト)を
自己同期的に判定できるという性質と、overlong 表現を悪用したバイト列偽装(NUL バイトインジェクション等)を
構造的に防ぐという、二重の安全性を担っているためである。Sign はこの安全性を毀損しない。

> [!IMPORTANT]
> **Sign における文字の「boxing」とは、標準 UTF-8 の先頭バイトが本来持つビットパターン
> (`0xxxxxxx` / `110xxxxx` / `1110xxxx` / `11110xxx`)そのものを指す。**
> 型情報(何バイト列か)は、UTF-8 自身の自己記述性から無償で得られる。
> MUTF-8 のような独自エンコーディングは必要ない。

---

## 2. UTF-8 エンコーディング規則(標準準拠)

| コードポイント範囲 | バイト列 | 先頭バイト |
| --- | --- | --- |
| U+0001 〜 U+007F | `0xxxxxxx` | 1バイト(ASCII 互換) |
| U+0080 〜 U+07FF | `110xxxxx 10xxxxxx` | 2バイト |
| U+0800 〜 U+FFFF(サロゲート域を除く) | `1110xxxx 10xxxxxx 10xxxxxx` | 3バイト |
| U+10000〜U+10FFFF | `11110xxx 10xxxxxx 10xxxxxx 10xxxxxx` | 4バイト |

> [!NOTE]
> **U+0000 は Char 型のコードポイント範囲に含まれない。** 理由は次章。
> **U+D800〜U+DFFF(サロゲート域)も同様に Char 型の範囲外**とする。単独の Scalar Value として
> 不正であることは Unicode 標準自身が規定しており、Sign 独自の制約ではない。

---

## 3. `__`(Unit)と U+0000 の関係:niche による統合

### 3.1 基本方針

Char 型の内部表現(コードポイントを保持する整数)は、Unicode の仕様上 **U+0000 を正当な文字として
使う理由が実務上ほぼ存在しない**(制御文字としての原義は失われて久しく、テキストの中身として
意味を持つ場面がない)。そこで Sign は、この 1 点を **Char 型の値域から構造的に除外する(niche)**。

```
Char 型の値域 := { U+0001, ..., U+D7FF } ∪ { U+E000, ..., U+10FFFF }

除外される 1 点 U+0000 のビットパターンは、
Char 型の文脈においては `__`(Unit)として扱われる。
```

これは Rust の `char` 型がサロゲート領域を除外し、`Option<char>` が `char` と同一サイズで
済むようにしている niche 最適化と、同型の技法である。

### 3.2 なぜこれで overlong 問題が消えるか

MUTF-8 は「U+0000 を無理に 1 コードポイントとして表現し続けるために、regular UTF-8 が禁止する
overlong 列を特例的に許可する」という設計だった。これは規格の穴を突く回避策であり、
標準準拠の UTF-8 デコーダを通過した瞬間に破綻する(不正列として拒否される、あるいは
過去に実際に悪用されたのと同じ攻撃面を再現する)。

Sign の設計はこれと根本的に異なる。**U+0000 という値そのものを Char 型の世界から消す**ため、
「U+0000 をどう安全にエンコードするか」という問い自体が発生しない。エンコードすべき対象が
存在しないのだから、特殊なバイト列も要らない。

### 3.3 `$__` の指すもの

```
Sign rodata(コンパイラ管理):
┌──────────────────────┐ ← $__ が指すアドレス
│ [Char niche スロット] │ (Char 型内部表現において構造的に出現しない値)
└──────────────────────┘
```

- 物理番地 0x00 とは無関係(ターゲットのメモリマップ上の意味は問わない)
- ARM のリセットベクタ・stdin(fd=0)・RISC-V の実装依存リセットベクタ等、いかなる
  ハードウェア規約とも衝突しない(niche はあくまで Char 型の内部表現空間の話であり、
  アドレス空間の話ではないため)
- 書き込み不可(read-only)。`$__ # X` はこの領域への書き込みであり、致命的エラーとする

## 4. Byte 型との明確な分離

バイナリセーフ性(任意の 0x00〜0xFF を正当なデータとして扱う必要)が要求される場面は、
Char 型ではなく **Byte 型** が担う。

| | Char 型 | Byte 型 |
| --- | --- | --- |
| 値域 | Unicode Scalar Value(U+0000, サロゲート域を除く) | 0x00 〜 0xFF 全域 |
| niche | あり(U+0000 が `__` と同一視される) | なし |
| 終端表現 | 不要(`__` は niche、センチネル不要) | 長さの外部化(`[len ~bytes]`)を強制 |
| 典型用途 | 文字列・テキスト | 画像・シリアライズ済みデータ・FFI 境界 |

Byte 型には niche が存在しないため(全 256 値が正当なデータになりうる)、終端やUnitを
バイト値に同居させる回避策は取らない。これは前々節の C 言語 NUL 終端・EOF センチネルが
抱えていた病理(データと境界情報を同一チャネルに多重化する誤り)を、構造的に再発させない
ための明示的な設計判断である。

外部から届く任意のバイト列(他システムが生成した、U+0000 を含みうる UTF-8 データ等)は、
まず Byte 型として受け取り、Char 型列への変換時に U+0000 の有無を検査する。この変換境界は
`@`(前置・input)が Unit で吸収元に倒れる規則と対応する。

## 5. Id射(`!__`)の dispatch

### 5.1 Id射はエンコード領域に格納しない

Id射の振る舞いは boxing されたバイト列の中身とは無関係であり、**コンパイラが静的に生成する**
(ゼロコスト)。

```
__ | X のコンパイル方式(概念):
  左辺が Unit → 右辺をそのまま返す(Id射の等価操作)
  左辺が非Unit → 左辺をそのまま返す(短絡評価)
```

### 5.2 `!__` の実体

`!__` は「Unit を受け取り引数をそのまま返す純粋関数」としてコンパイラが生成する関数オブジェクト。
型は `∀A. A → A`。`__`(零対象)とは異なる非 Unit 値(真)である。

## 6. I/O との衝突まとめ

| 値 | UTF-8 表現 | 生の 0x00 | ARM ベクタ衝突 | stdin(fd=0)衝突 |
| --- | --- | --- | --- | --- |
| `__`(Char 文脈) | (存在しない、niche) | なし | なし | なし |
| `"hello"` | 標準 UTF-8 準拠 | なし(Char 型に U+0000 が存在しないため) | なし | なし |
| 任意バイナリ | Byte 型(長さ外部化) | あり得る(正当なデータとして) | 該当なし(Byte型はアドレス空間と無関係) | 該当なし |

> [!IMPORTANT]
> Sign の文字列は標準 UTF-8 として出力され、**外部の任意の標準準拠 UTF-8 ツール
> (OS・ブラウザ・他言語ランタイム等)でそのまま安全に解釈できる。** MUTF-8 のような
> Sign 内部限定の安全性ではなく、業界標準への準拠そのものによる安全性である。
