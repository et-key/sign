# System Semantics (Sign言語 システム記述仕様)

## 概要
Sign言語は「高水準な関数型記述（アプリケーション層）」と「低水準なプリミティブ操作（システム層）」を同一の構文でサポートします。
本ドキュメントでは、特にシステム層（コンパイラ、アロケータ、ドライバ等）の実装において使用される**明示的なメモリ操作**について定義します。

## アプリケーション層とシステム層の違い

### アプリケーション層 (ユーザーコード)
*   **メモリ管理**: 自動（コンパイラ/GCが隠蔽）。`alloc` などの直接呼び出しは行わない。
*   **値の扱い**: 参照渡し・値渡しはコンテキスト（関数適用）によって自動的に最適化される。
*   **アドレスの直接操作**: `0x00 # 1` `@0x00` のような直接的なメモリアクセスが可能。
*   **禁止事項**: ポインタのインクリメント、デクリメント。

### システム層 (関数ブロック内部)
*   **メモリ管理**: 明示的。`alloc` 相当のロジックを構成可能。
*   **明示的な参照操作**: `$` (Address Of), `@` (Deref), `#` (Store) を使用して、物理的なメモリ配置やポインタの受け渡しを制御する。

## 明示的な参照操作 (Explicit References)

システム記述（特にアロケータや高効率なライブラリ機能の実装）において、値(Value)と参照(Reference/Address)を区別するために以下の演算子を使用します。

### 1. `$` Reference (Address Of)
変数の「値」ではなく「場所（アドレス）」を取得します。
主に関数へ「参照渡し」を行いたい場合や、変数のスロットそのものを操作したい場合に使用します。

```sign
map $[* 2] ...  ` 関数オブジェクトを値コピーせず、参照として渡す
$heap_ptr # ... ` heap_ptr変数そのもの（スロット）に書き込む
```

### 2. `@` Dereference (Load)
参照（アドレス）から実体（値）を取り出します。
参照渡しされた引数を利用する際に使用します。

```sign
map : f ... ?
    @f ...      ` 参照 f から関数実体を取り出して適用
```

### 3. `#` Store (Update)
参照先（アドレス）に値を書き込みます。
変数の更新（再代入）や、ヒープメモリへの書き込みに使用します。
左辺は必ずアドレス（L-Value）でなければなりません。

```sign
$heap_ptr # next ` heap_ptr変数のアドレス($)を取得し、そこに next を書き込む
```

## コンパイラの挙動 (Proto A5)

*   **識別子 (Identifier)**: 常に **R-Value (Value)** として評価されます。
    *   `x` -> `ldr x0, [x_addr]`
*   **代入文脈 (# の左辺)**: 特別扱いはしません。左辺はアドレスとして評価される式でなければなりません。
    *   `$x # y` -> `adr x0, x_addr; ... str y, [x0]`
        `x`のアドレスを取得し、その指す先へ`y`を書き込む（`*x = y`）
*   **明示的な操作**: `$` や `@` を全面的に使用し、マジック（暗黙の型変換）を排除します。

## 参照演算におけるリテラル（`0x00`, `0u00`, `__`）の振る舞い

型システムと Boxing（MUTF-8 等）の仕様に基づき、これらのリテラルに対する参照操作は明確に区別されます。

### 1. `$` (Address Of) の挙動

`$` は定数リテラルに対しても適用可能であり、コンパイラが静的に確保した `.rodata` 領域等のアドレス（Read-Only）を返します。

*   **`$0x00`**: コンパイラが確保した **Int(0) のスロット** のアドレスを返す。
*   **`$0u00`**: コンパイラが確保した **MUTF-8 (`0xC0 0x80`) のスロット** のアドレスを返す（例: `0x200` などのデータセグメント開始位置）。
*   **`$__`**: コンパイラが確保した **Unit マーカーのスロット** のアドレスを返す。

これらはすべて異なる有効なアドレスであり、物理番地 `0x00` を指すわけではありません。
（※これらに `#` で書き込もうとした場合は致命的エラーとなります）

### 2. `@` (Dereference) の挙動

`@` は右辺をアドレス（L-Value）として解釈し、そのメモリを読み取ります。

*   **`@0x00`**: Int型の `0` を「アドレス」として解釈し、**番地 0x00 から値を読み取る**。
*   **`@0u00`**: Char型の値をアドレスとして解釈しようとするため、**型エラー** または **非対称 Unit 伝播により即座に `__`** に崩壊する。
*   **`@__`**: 左辺が Unit であるため、式全体が即座に **`__`（吸収元）** に崩壊する。

## ターゲット層 (Layer) による物理アドレス参照の差異

`@0x00` や `0x00 # expr` のような「Int値を用いた直接メモリアクセス」の挙動は、実行ターゲットの `layer`（`Option_MS_Spec_ja-jp.md` 参照）に依存します。

### Layer 0 (Baremetal / MCU / Kernel)
*   **`@0x00`**: 物理メモリアドレス `0x0000` を直接読み取ります（ARM のリセットベクタ、割り込みベクタテーブル等）。
*   **`0x00 # expr`**: 物理番地 0 への直接書き込みを実行します。

### Layer 1〜4 (OS上のユーザ空間 / VM / JS)
*   **`@0x00`**: 仮想アドレス `0x00` へのアクセスとなり、大半の OS では **Segmentation Fault**（Null Pointer Dereference）を発生させます。
*   システム層記述として例外ハンドラでトラップされるか、プロセスが即座に終了します。
*   安全性を重視するレイヤーでは、このような未保護のメモリアクセスは事前にバリデーションされるか、意図的なクラッシュ（フェイルファスト）として機能します。
