# Sign vs C: セキュリティ比較

> このドキュメントは実験的・考察段階の内容です。

## Sign は C よりセキュアになる

これは設計上の帰結として自明である。

### C の主要な脆弱性クラスと Sign での扱い

| C の脆弱性 | 発生原因 | Sign での扱い |
|-----------|---------|--------------|
| バッファオーバーフロー | 配列境界チェックなし | alloca の連続ブロックはサイズが静的確定 → 境界はコンパイル時に既知 |
| Use-after-free | malloc/free の手動管理 | ヒープがない（layer: 0〜3）→ free 自体が存在しない |
| Null 参照クラッシュ | NULL ポインタの逆参照 | `__`（Unit）が伝播するだけ → クラッシュしない、例外も投げない |
| 未初期化 FPU の使用 | 起動直後に float 演算 | `layer < 2` でコンパイルエラー → 静的に防止 |
| 型の混同（void\* / union） | 型パニングの許容 | 構造的型付けで型混同が起きない |
| 二重解放（double-free） | 手動 free の多重呼び出し | alloca はスコープ終了で自動回収 → free 操作なし |
| 整数オーバーフロー | 暗黙の型変換 | `Int` / `Float` の明示分離、暗黙変換なし |

C における脆弱性の大部分は、コンパイル時に検出可能だった問題を実行時まで持ち越したことが原因である。
Sign はこれらを**構造的に起き得なくする**か、**コンパイルエラーとして検出する**。

---

### ファームウェア層でのインパクト

BIOS / UEFI の脆弱性は、一般的な OS 層の脆弱性より深刻である：

- **Ring-0 より深い** — OS のセキュリティ機構が届かない
- **永続化が容易** — ファームウェアに潜めば OS 再インストールでは除去できない
- **署名検証を潜り抜ける可能性** — ブートチェーンの最初期に存在する

Sign で書かれた AI 生成の BIOS コードは：
1. Use-after-free / バッファオーバーフロー が構造的に存在しない
2. FPU 未初期化状態での演算がコンパイルエラー
3. AI による生成でヒューマンエラーが排除される

という三重の安全性を持つ。
