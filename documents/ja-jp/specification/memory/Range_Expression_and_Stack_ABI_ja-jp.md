# レンジ式とスタックABI仕様

この文書は、Sign言語における**レンジ式（Range Expression）**の構文・意味論と、
**スタックベースのカリー化ABI**について定義する。

---

## 設計原則：静的機能のみ

SignはCと同様に、**コンパイル時に決定できる静的な機能のみ**を言語仕様として保証する。

- 動的ディスパッチ・遅延評価・GCは言語の責務ではない
- 境界が静的に決まる構造はコンパイラが完全に解析できる
- 動的な振る舞いが必要な場合はユーザーが上のレイヤーとして積む

これにより、Sign自体はゼロコスト抽象化の静的レイヤとして機能する。

### 先行プロジェクトとの比較

同様の思想に立つ先行プロジェクトとして、Static TypeScript（AssemblyScript）および rPython がある。
Signはこれらと設計軸を共有するが、既存言語のサブセットではなく**最初から制約込みで設計された独立言語**である。

| 観点 | C | Static TypeScript | rPython | Sign |
|------|---|------------------|---------|------|
| 起源 | 独自設計 | TypeScriptのサブセット | Pythonのサブセット | **独自設計** |
| 主な用途 | システム・組み込み | WASM・組み込み | JIT生成器 | **低レイヤ・セキュア** |
| 静的型のみ | ✓ | ✓ | ✓ | ✓ |
| GCなし | ✓ | ✓ | ✗（独自GC） | ✓ |
| ヒープ管理 | 手動（malloc/free） | 部分的に手動 | RPython GC | **allocaファースト** |
| クロージャコスト | 手動（関数ポインタ） | あり | あり | **スタックのみ** |
| 型システム | 構造（弱い） | 構造的 | フロー解析 | **圏論的** |
| ループ表現 | `for`/`while` 文 | `for`/`while` 文 | `for`/`while` 文 | **レンジ式（返値あり）** |
| 親言語の負債 | なし | あり（TSの動的機能） | あり（Pythonの動的機能） | **なし** |

**Signの優位点**：親言語の設計負債がないため、静的制約を構文レベルから一貫して設計できる。
`alloca` ファーストのメモリモデルとレンジ式によるループの式化は、
C相当のパフォーマンスを保ちながら安全性と表現力を高める独自の解法である。

## 1. レンジ式（Range Expression）

### 1.1 基本構文

```
[start ~ end]
```

**「返値を持つfor式」**として定義される。`for` キーワードは存在しない。
レンジ式自体が**関数**であり、評価されると値を返す。

```sign
` 1から5まで
[1 ~ 5]   ` → 1, 2, 3, 4, 5

` インデックス変数に束縛する
i : [1 ~ 5]
```

### 1.2 ステップ付きレンジ

```
[start ~+ step ~ end]
```

ステップを明示したレンジ。ストライドアクセスパターンが構文に現れるため、
コンパイラがSIMD化・ループアンロールの判断に必要な情報を静的に持つことができる。

```sign
[0 ~+ 2 ~ 10]   ` → 0, 2, 4, 6, 8, 10
```

### 1.3 開端レンジ（カウンタ）

```
[start ~+ step]
```

上界を持たない開端レンジ。Cの無限ループカウンタ（`int c = start; ...`）に相当する。
**終了条件は呼び出し元の責任**であり、言語は保証しない（Cと同じ立場）。

```sign
` 0から始まる無限カウンタ
[0 ~+ 1] 
```

## 2. レンジ式による要素アクセス（`'`）

`'` 演算子（get）はコレクションから要素を取り出す。
インデックスとしてレンジ式を使うことで、スライス操作を表現できる。

```sign
[0 ~+ 2 ~ 10] ' [1 ~ 3]
```

```
[0 ~+ 2 ~ 10]  →  [0, 2, 4, 6, 8, 10]
                    ↑  ↑  ↑  ↑  ↑  ↑
インデックス:        0  1  2  3  4  5

' [1 ~ 3]  →  インデックス 1, 2, 3 の要素を取る

結果: [2, 4, 6]
```

イテレーションの記述とスライスの記述が**同一の構文**で統一される。

## 3. レンジ式を用いたループ

### 3.1 ラムダへの適用

レンジ式は高階関数として `$[...]` ブロック（ラムダ）を受け取る。

```sign
[0 ~+ 2 ~ 10] $[
    i ?
    ` i を使う処理
]
```

- `[0 ~+ 2 ~ 10]` → イテレータを生成する関数に見えるが、forの記述と相違ない。
- `$[ i ? ]` → `i` を引数に取る無名関数に見えるだけで、実態は単なるforの実行ブロック
- 並置（関数適用）= ループ

実体は結果を返す**for文**である。

### 3.2 コンパイル後の形式

コンパイラはこの構造をタイトなアセンブリループに変換する。
`$[...]` のプロローグ/エピローグはインライン展開前提の呼び出し規約に従い、
関数呼び出しのオーバーヘッドはゼロになる。

```asm
; [0 ~+ 2 ~ 10] $[ i ? ... ]
    mov  rax, 0       ; i = 0（開始値）
.loop:
    ; ... $[...] の本体（インライン展開）...
    add  rax, 2       ; ステップ
    cmp  rax, 10
    jle  .loop
```

### 3.3 無限ループとカウンタ

開端レンジ `[0 ~+ 1]` を関数の引数として使うことで、再帰の表現が関数の再帰ではなく単なるループにしたループカウンタを表現できる。

```sign
f :
    c : [0 ~+ 1]
  ?
    ` c を使った処理
```

このパターンは：
- 構文上は再帰に見えない
- コンパイラが `c : [1 ~]` を認識した時点で「ループカウンタ」と判断
- 実際には `inc rax` + `jmp .loop` のタイトなループになる

**再帰という概念を使わずに、純粋なループを記述できる。**

## 4. スタックベースのカリー化ABI

### 4.1 設計方針

Signは**すべてのプロジェクトデータをスタック（alloca）で管理する**ことを基本方針とする。

Cがヒープ（`malloc`/`free`）を多用する結果として生じる問題：
- use-after-free
- ヒープオーバーフロー
- メモリリーク

これらをSignはデフォルトで回避する。
`alloca` によるスタック確保はスコープ終了と同時に自動回収されるため、
GCも所有権注釈も不要である。

### 4.2 カリー化クロージャのスタック実装

通常、カリー化（部分適用）のクロージャはヒープに確保される：

```c
// 一般的な関数型言語ランタイム
closure* c = malloc(sizeof(closure));  // ← ヒープ
c->fn = f;
c->captured_arg = a;
```

Signは代わりに **`alloca` によるスタック上のリンクドリスト** を使用する：

```
f a      →  alloca node {val: a, next: NULL}       ← 第1引数
f a b    →  alloca node {val: b, next: →前のノード}  ← 第2引数
f a b c  →  全引数が揃った → 実行 → スタック一括回収
```

- 部分適用のたびにスタックフレームが1ノード積まれる
- ヒープ確保は一切発生しない
- スコープを抜ければリンクドリスト全体が自動消滅

### 4.3 引数順序の保証とコンビネータ規約

引数の順序はコンビネータの規約に従い**静的に保証**されなければならない。
リンクドリストは挿入順（FIFO）で保持し、呼び出し時にその順でレジスタに積む。

```
リスト走査: a → b → c
レジスタ割り当て: a→rdi, b→rsi, c→rdx   (System V ABI 例)
```

これにより、Sコンビネータ等の変換においても引数順序の整合性が保たれる。

### 4.4 スタック上のキューセマンティクス

`alloca` リンクドリストの重要な性質：

- **スタック**は物理的にLIFO（Last In, First Out）
- **リンクドリスト**は論理的にFIFO（First In, First Out）を実現

この組み合わせにより、**ヒープを一切使わずキューセマンティクスを実装できる**。

```
物理スタック（LIFO）:
  [ node_c ]  ← スタックトップ
  [ node_b ]
  [ node_a ]  ← スタックボトム

論理走査（FIFO）:
  node_a → node_b → node_c  ← 引数順序で走査
```

これはカリー化だけでなく、引数順序が重要なあらゆる場面での
スタックベース処理に応用可能である。

### 4.5 静的アリティの前提

このABIが成立するのは：

- 関数のアリティ（引数の数）が**コンパイル時に確定している**こと
- リンクドリストのノードサイズが**静的に計算可能**であること

のいずれも静的設計原則（§設計原則）から保証される。

## 5. まとめ

| 機能 | Cの相当物 | Signの表現 | コスト |
|------|-----------|-----------|--------|
| for ループ | `for(int i=0; i<=5; i++)` | `i : [1 ~ 5]` | ゼロ（インライン展開） |
| ストライドアクセス | `for(int i=0; i<=10; i+=2)` | `[0 ~+ 2 ~ 10]` | ゼロ（SIMD対象） |
| 無限ループカウンタ | `int c=0; while(1){c++;}` | `c : [0 ~+ 1]` | ゼロ（レジスタのみ） |
| スライス | `arr[1], arr[2], arr[3]` | `arr ' [1 ~ 3]` | インデックス走査のみ |
| クロージャ | `malloc(sizeof(closure))` | `alloca` 上のリンクドリスト | スタックのみ |
| FIFO キュー | `malloc` ベース連結リスト | `alloca` 上のリンクドリスト | スタックのみ |


Signの設計は「高レベルな構文に見えるが、コンパイル後はCと同等のコスト」を目指す。
構文と実装コストを切り離し、ABIの規約と静的解析によってゼロコストを実現する。

## 6. alloca の2用途：リンクドリスト vs 連続ブロック

`alloca` によるスタック確保は、Sign内で**2つの異なる目的**に使われる。
混同しないよう、用途を明確に区別する。

### 6.1 リンクドリスト（カリー化専用）

カリー化（部分適用）の引数キャプチャに使用。§4 で説明した通り。

```
用途: 関数 f a b c の引数チェーン
構造: node_a → node_b → node_c（ポインタで連結）
確保: alloca(sizeof(ArgNode)) × アリティ回
特性: FIFOキューセマンティクス、引数順序を保証
```

**Arrayには使えない理由**：ポインタ追跡が発生するため、
ランダムアクセス O(n)、SIMDロード不可、キャッシュ効率が低い。

### 6.2 連続ブロック（Array / Vector 専用）

`List` / `Vector` 型の実体として使用。要素がメモリ上で連続している必要がある。

```
用途: [1 2 3 4 5] のような配列データ
構造: 要素が連続したメモリブロック
確保: alloca(sizeof(elem) × n)
特性: ランダムアクセス O(1)、SIMDロード可能、キャッシュ局所性が高い
```

```c
// Sign コンパイラが生成するコード（概念）

// カリー化引数 → linked list
ArgNode* node = alloca(sizeof(ArgNode));   // ポインタで連結

// List [1 2 3 4 5] → 連続ブロック
int* arr = alloca(sizeof(int) * 5);        // 連続メモリ
arr[0] = 1; arr[1] = 2; /* ... */
```

### 6.3 使い分けの判断基準

| データの種類 | alloca の用途 | 理由 |
|-------------|--------------|------|
| カリー化の引数 | リンクドリスト | アリティが静的、順序保証が必要 |
| `List` / `[1 2 3]` | **連続ブロック** | O(1)アクセス、SIMD対象 |
| `Vector` (SIMD型) | **連続ブロック** | SIMDロード命令が連続メモリを要求 |
| レンジ式の一時バッファ | 連続ブロック | stride アクセスのベクトル化に必要 |
| DMA・ハードウェアバッファ | 連続ブロック | ハードウェアが連続アドレスを要求 |

### 6.4 レンジ式とベクトル化の前提

レンジ式 `[0 ~+ 2 ~ 10]` に `layer: 3` 以上で SIMD 最適化を適用するには、
実体が連続ブロックであることが前提：

```sign
` layer: 3 以上
` [0 ~+ 2 ~ 10] の実体が連続ブロックなら、コンパイラは SIMD gather/scatter を生成できる
[0 ~+ 2 ~ 10] ' [1 ~ 3]   ` → SIMD gather 命令に変換可能
```

リンクドリストで実装された場合、このベクトル化は不可能になる。

## 7. 構造体と多次元配列の alloca 表現

### 7.1 構造体（Struct / Dict）

`Dict`（`[key : val]` 記法）の実体は、フィールドが**固定オフセットで並ぶ連続ブロック**である。
コンパイラはフィールドの型・サイズ・アライメントを静的に決定し、単一の `alloca` で確保する。

```sign
` point : [x : 3  y : 4]  のコンパイル後のイメージ（概念）
point : [
    x : 3
    y : 4
]
```

```c
// コンパイラが生成するコード（概念）
struct { int x; int y; } *point = alloca(sizeof(*point));
point->x = 3;
point->y = 4;
// スコープを抜けると自動回収（free不要）
```

**フィールドのlayer制約**：フィールドの型にも layer 制約が適用される。
`Float` フィールドを持つ Struct は `layer: 2` 以上でのみ使用可能。

### 7.2 多次元配列

多次元配列は**行優先（row-major）の連続ブロック**として alloca で確保する。

```
2次元 [rows][cols]:
  確保: alloca(sizeof(elem) * rows * cols)
  アクセス: element[i][j] = *(base + i * cols + j)
```

サイズは**コンパイル時に静的に決定**されている必要がある（静的設計原則より）。

### 7.3 `'` の連鎖による多次元スライス

多次元配列へのアクセスは、`'`（get）演算子を**左結合でチェーン**することで表現する。
特別な多次元構文は不要。

```sign
` 3×5行列 matrix の行1〜2、列0〜3 を取り出す
matrix ' [1 ~ 2] ' [0 ~ 3]
```

**評価の順序：**

```
① matrix ' [1 ~ 2]
     → 行1〜2 を取り出す（部分行列へのビュー）
     → ポインタ計算のみ（ゼロコピー）

② (結果) ' [0 ~ 3]
     → 列0〜3 を取り出す
     → さらにポインタ計算のみ
```

**n次元への拡張：**

```sign
` 3次元テンソルのスライス
tensor ' [1 ~ 2] ' [0 ~ 3] ' [2 ~ 4]

` 特定の要素アクセス（スカラー）
matrix ' 2 ' 3   ` → matrix[2][3] と同等
```

### 7.4 スライスのゼロコピー性

`'` によるスライスはポインタ計算のみで実現される。コピーは発生しない。

```
matrix ' [1 ~ 2] の実装:
  result_ptr  = base + 1 * num_cols * sizeof(elem)
  result_len  = 2 * num_cols
  ↑ これだけ（メモリコピーなし）
```

`layer: 3` 以上では、スライス後の連続領域に SIMD 命令を直接適用できる：

```sign
` layer: 3 以上
` 行1〜2のデータをSIMDで処理
matrix ' [1 ~ 2] ' [0 ~ 4] $[
    row ?
    ` row は連続メモリ → SIMD命令に変換可能
]
```

### 7.5 データ種別と alloca 戦略まとめ（更新版）

| データの種類 | alloca の形式 | アクセス | SIMD対象 |
|-------------|--------------|----------|----------|
| カリー化の引数 | リンクドリスト | 順次走査のみ | ✗ |
| `List` / 1次元配列 | 連続ブロック | O(1) | ✓ |
| `Vector` (SIMD型) | 連続ブロック | O(1) | ✓ |
| `Struct` / `Dict` | 連続ブロック（固定オフセット） | フィールド名→オフセット | △ |
| 多次元配列 | 連続ブロック（row-major） | `'` チェーン | ✓ |
| DMA・ハードウェアバッファ | 連続ブロック | 直接アドレス | ✓ |
