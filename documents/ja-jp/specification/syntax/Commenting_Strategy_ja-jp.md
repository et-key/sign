# Sign言語のコメント規則

## 基本原則

Sign言語のコメントは**先読みルール**によって文字列リテラルと区別されます。

```
バッククォート ` を見つけた後、同じ行に閉じバッククォートがあるか確認する:
  - 閉じあり → 文字列リテラル（どこに現れても）
  - 閉じなし（EOLまで） → コメント
```

> [!IMPORTANT]
> **行頭に `` ` `` が現れた場合も同じルールが適用される。**
> 閉じバッククォートがなければコメント、あれば文字列リテラル。

```sign
`これはコメント（閉じなし → EOLまでコメント）
`main.sn`@~    ← 閉じあり → `main.sn` は文字列リテラル（インポートパス）
x : `Hello`    ← 閉じあり → 文字列リテラル
```

---

## ルール定義

```
comment = SOL ` [^\n\r`]* EOL    ← 閉じ ` がない → コメント（行頭・行中どこでも）
string  =     ` [^\n\r`]* `      ← 閉じ ` がある → 文字列リテラル
```

**詳細**: `specification/syntax/String_and_Comment_ja-jp.md` を参照。

---

## 具体例

```sign
`これはコメント（行頭・閉じなし）
` これもコメント（行頭・閉じなし）

x : `Hello, World!`   ` ← x に文字列を束縛するコード行
                       ` ← この部分がコメント（閉じなし）

`path/to/file`@~      ← 行頭だが閉じあり → `path/to/file` は文字列（インポート）
```

## インデント内のバッククォート

インデント（タブ）の後に `` ` `` が来る場合は、行頭ではないのでコメントにならない。
先読みルール（閉じあり=文字列、閉じなし=コメント）が適用される。

```sign
calc_func : x ?
\t`ドキュメント文字列（タブ後・閉じあり → 文字列として評価されるが捨てられる）`
\tx * 2
```

この文字列は評価されますが戻り値として使われないため、事実上コメントとして機能します。

## 文法仕様

- 字句解析レベルでコメント（閉じなしバッククォート行）を削除しても構いません
- シンタックスハイライトやLanguage Serverは**「行頭から始まり閉じバッククォートがない `` ` `` 行」**をコメント色で表示することが推奨されます
- PEGでは先読み（lookahead）で閉じバッククォートの有無を判定します
