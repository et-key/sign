# 関数定義とその型について

Signでは、関数定義の方法が多岐にわたって存在する。  
よって、これら関数定義の方法について、具体例を交えたドキュメントをここに記す。

## ポイントフリー記述

全ての演算子を関数として扱う場合、任意のカッコで演算子を囲むことで関数として扱う。  
この時、前置演算子は、`[<op>_]` 後置演算子は `[_<op>]`と記述する。  
ポイントフリー記述の二項演算子は、複数の引数を貪欲に演算する。  
単項式の後ろに`,`を付けたポイントフリー記述は、そのすべてに適用される。

```Sign

` Sumと同等
[+] 1 2 3 4 5

` mapと同等
[* 2,] 1 2 3 4 5

` 否定する関数
[!_] 2 < 3

` 階乗する関数
[_!] 5
```

## ? 演算子を用いた定義方法

`?`演算子は、左辺に引数リスト、右辺に本体を有する演算子と解釈できる。

```Sign

f : x y ? x + y
```

### match_caseの記述

`?` の右辺を改行・インデントブロックを挟むことで、本体内の : 演算子はmatch_caseとなる。

```Sign

f : x y ?
	x > 3 : x - y
	y < 3 : x + y
	x y
```

### デフォルト引数の記述

`?` の左辺インデントブロックは : 演算子がデフォルト引数の記述となる。

```Sign

f :
		x
		y : x + 1
		z : y + 1
		~rest
	? x y z rest~
```

> [!IMPORTANT]
> **デフォルト引数には純粋な値式のみ使用可能。IO 演算子（`@`・`#`）は禁止。**
>
> デフォルト引数内に `@`（デリファレンス）や `#`（ストア）を記述することはできない。
> デフォルト引数はコンパイル時に特殊化される純粋な値として扱われるため、
> 実行時 IO を含む式とは整合しない。IO が必要な処理は関数本体（`?` の右辺）に記述すること。
>
> ```sign
> ` ❌ 禁止：デフォルト引数内での IO
> f :
>     x : @some_ptr   ` ← コンパイルエラー
>   ? x
>
> ` ✅ 正しい：IO は本体に記述
> f :
>     x
>   ? @some_ptr
> ```

### デフォルト引数と、match_caseの記述は共存可能。

```Sign

f :
		x
		y : x + 1
		z : y + 1
		~rest
	? 	x > 3 : x - y rest~
		y < 3 : x + y rest~
		z rest~
```

これらの関数にそれぞれ型を与えた方がよい。

## 関数適用時

デフォルト引数を持つ関数の適用だけが異なる挙動となる。

```Sign

` 通常は部分適用となる動作（静的脱糖によりラムダに変換される）
f : [+] _ _ 3 4 5  ` $p0 $p1 ? [+] $p0 $p1 3 4 5 と同等
f 1 2              ` 評価結果: 15

g :
		x
		y : x + 1
		z : y + 1
	? x y z

` g はデフォルト引数を持つため、これらはアリティ計算から除外される。
` よって、デフォルト引数以外の実引数を渡すだけで自動的に評価が実行される。
g 3   ` 評価結果: 3 4 5

` 例えば、予期せぬ演算で __（Unit値）が算出された場合、デフォルト引数を持たない関数は __ に短絡評価される。
` （Unit伝播・即時崩壊）
f 3 < 2 1   ` 3 < 2 が __ になるため、f __ 1 となり、評価結果は __ に崩壊する

` デフォルト引数を持つ関数の引数に __ が渡された場合、暗黙でデフォルト引数の定義値による評価が行われる（フォールバック）。
g 1 3 < 2   ` 3 < 2 が __ になるため、g 1 __ となり、y にはデフォルト値 x + 1 (2) が適用される。
            ` 評価結果: 1 2 3
```

## カッコ囲みの仮引数リストによるチルダ省略

仮引数リスト全体をカッコで囲んで定義した関数は、実引数として渡されたリストや構造体を、呼び出し側で後置チルダ `~`（展開）を付与することなく、そのまま直接受け取ることができます。
この構文の場合、参照が2つ以上あっても参照の数だけカッコで囲むことで、参照の展開による境界を明示することが出来ます。
もちろん、デフォルト引数やmatch_caseとの共存も可能です。
```Sign
` 仮引数リストをカッコで囲む
sum_list : [x ~xs] ? xs & x + sum_list xs | x

` 呼び出し側はチルダなしでリストを渡せる
sum_list [1 2 3 4 5]  ` 評価結果: 15

` デフォルト引数・match_caseとの共存例
func_mixed :
		[
			x
			y : x + 1
			~z
		]
	?
		x > 3 : x - y
		y

func_mixed [5]      ` 評価結果: -1 (x=5, y=6. x > 3 なので 5 - 6 = -1)
func_mixed [2]      ` 評価結果: 3  (x=2, y=3. x > 3 は偽(__)のため、y が評価される)
func_mixed [2 10]   ` 評価結果: 10 (x=2, y=10. x > 3 は偽(__)のため、y が評価される)
```

## 構造体メンバーの一致による自動バインディング

仮引数リスト全体が `[ ]` で囲まれている関数に構造体（辞書）を実引数として渡す際、構造体のメンバー名（キー）と関数の仮引数名が一致している場合、順番に関わらず自動的に名前ベースで分割代入（バインド）が行われます。
この構文の場合、参照が2つ以上あっても参照の数だけカッコで囲むことで、参照の展開による境界を明示することが出来ます。
もちろん、デフォルト引数やmatch_caseとの共存も可能で、Listの時の仮引数リストでの展開と同様の構文で定義できます。

```Sign
` 構造体を受け取る関数、~obj は必須ではないが、定義時に渡した構造体以下の構造体を保持したい場合に使う。
calc_diff : [foo bar ~obj] ? foo - bar

` メンバー名を一致させた構造体をそのまま（チルダなしで）渡す
calc_diff [
	bar : 20
	foo : 100
]
` 評価結果: 80 (foo = 100, bar = 20 が自動で紐づく)

func_mixed :
		[
			foo
			bar : foo * 2
			~obj
		]
	?
		foo > 3 : foo - bar
		bar

func_mixed [
	foo : 4
]
` 評価結果: 8 (foo = 4, bar = 8. foo > 3 は真のため、foo - bar が評価される)

```
