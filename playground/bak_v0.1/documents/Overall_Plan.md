# 新生Sign WASMコンパイラ 全体実装計画

## 1. 開発のコア・フィロソフィー
Sign言語は「ハードウェアを抽象化するのではなく、型と言語を抽象化する」という哲学（Zero Cost Domain Abstraction）を持ちます。WASM版は**WASMの流儀（スタックマシン、ローカル変数、構造化制御フロー）**を最大限に尊重し、ネイティブなパフォーマンスと圏論的な美しさを両立させます。

## 2. WASMアーキテクチャの基本設計
段階的な実装アプローチをとりますが、最終的なゴールとして以下のモダンなコンパイラ設計を採用します。

* **NaN-Boxingと動的ディスパッチ**: すべてのスタック値は `f64` として扱います。数値以外の型（リストのポインタ等）はNaN空間にタグ付けし、実行時またはコンパイル時に演算子の振る舞いを振り分けます。
* **ローカル変数とエスケープ解析 (Ph5で本格稼働)**:
  * 変数は原則としてすべてWASMの `local` レジスタに割り当てます。
  * AST走査時に「アドレス取得（`$`）」や「クロージャのキャプチャ」を検知し、それらエスケープした変数のみを線形メモリ上に配置します。
* **真偽値モデル**:
  * Signでは `_` (Unit = WASM上では `NaN`) と 「未評価のラムダ項」 のみが Falsy（偽）です。
  * `0.0` を含むそれ以外は Truthy（真）です。WASMの比較命令が返す `0` は、必ず `NaN` にラップ・変換する処理を挟みます。

## 3. 「ブロック」の圏論的セマンティクス（絶対に守るべき罠）
Signにおいて、ブロック（複数行の式）はC言語のような単なる「dropの連続」ではなく、純粋関数型におけるモナドの結合（Monadic Bind）です。
* **空白（Space）**: 余積を通じた純粋な数学的合成（空間的並置）。
* **改行（Newline/Block）**: 関数合成がパイプラインのように連鎖する双代数的な構造（時間的並置）。
* **文脈依存の `:`**:
  * **関数ブロック**: `br_if` による早期リターン（ガード節）。
  * **定義ブロック**: 辞書型や let in のような束縛。
  * **式内部**: `if ... else ... end` を用いた値を残す三項演算子。

## 4. コンパイル・パイプライン
1. **AST正規化パス**: 糖衣構文の展開、ブロックのモナド結合の解決、`0b/0r`のRegister型統合。
2. **スコープ＆エスケープ解析パス**: 全変数のライフサイクル（localかメモリか）を決定。
3. **WAT生成パス**: 解析結果に基づくWASMアセンブリの出力。

## 5. 開発フェーズ（マイルストーン）
* **Ph1: 電卓**（リテラル、四則・論理・ビット演算、式内部のガード、単純なローカル変数定義）
* **Ph2: スタックマシンとしての充足**（ポイントフリー記法、関数/演算子の合成、リスト基盤の扱い）
* **Ph3: 関数ブロック**（`?` に続くブロック、`:` による早期リターンと構造化制御フローの実装）
* **Ph4: 定義ブロック**（複数式の手続き的/モナド的合成、辞書型、スコープ束縛）
* **Ph5: RAMマシンとしての駆動実装**（`@`, `#`, `$` 系による明示的なメモリアクセス、エスケープ解析、線形メモリへのアロケーション統合）
