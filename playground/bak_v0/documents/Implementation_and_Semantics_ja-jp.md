# Sign言語: 意味論とアセンブリ実装の分離（Zero Cost Domain Abstraction）

## 1. 概要と設計哲学
Sign言語のコンパイラ（WASM / AArch64）の実装において、**「ユーザーが記述するSign構文（圏論的意味論）の美しさを正とし、内部処理はそれを実現するための愚直なアセンブリに徹する」**という方針を明確に定義する。

Signは「C言語のパフォーマンスとPythonの記述性」の両立を目指しており、実行時の動的な型判定や重いランタイムを極力排除する（Zero Cost Domain Abstraction）。その過程で生じる「数学的な理想」と「ハードウェアの物理的挙動」の衝突は、コンパイラのレイヤーで吸収・分離して解決する。

## 2. ブロック逐次評価と MatchCase の同居
Signのブロック構文 `[...]` は、純粋な「辞書型の定義」と「モナディックな逐次処理」を同居させる。

* **ブロックの基本ルール**: 上から下へ式を逐次評価し、途中の値は破棄（スタックから `drop`）され、最後の式の値がブロック全体の戻り値となる。
* **Zero Cost MatchCase**: ブロック内で条件分岐を行う際、Signは予約語を持たず `:`（Guard）を使用する。
  コンパイラはASTを解析し、`:` の左辺が「識別子(名前)」であればメモリストア（変数定義）のアセンブリを展開し、「式(比較演算など)」であればハードウェアのジャンプ命令（WASMの `br` / AArch64の `B.cond`）を展開する。
  これにより、ユーザー視点では美しい宣言的パターンマッチに見えるが、内部的にはC言語の `if-else` や `goto` と等価なオーバーヘッドゼロのジャンプとして動作する。

## 3. インラインGuardとブロック脱出の文脈分離
MatchCase構文（`:`）が記述された「文脈」によって、コンパイラは脱出命令のスコープを切り替える。
* **ブロック直下の Guard** (`[ x = 0 : 100 ]`)：現在の親ブロックから即座に脱出（早期リターン）するパターンマッチとして機能する。
* **式内部の Guard** (`(x = 0 : 100)`)：脱出は行わず、単に値をスタックに残して次へ進む三項演算子として機能する。
これにより、意図しないスコープからの強制脱出（プログラムの早期終了）をコンパイラレベルで完全に防ぐ。

## 4. 真偽値とコントロールフロー(存在チェック)の分離
Signの圏論的真偽値モデルにおいて、「Unit(`_`) と未評価の関数(クロージャ)は False である」と定義されている。
しかし、この厳密な論理仕様をコントロールフロー（パイプラインやOR演算子のフォールバック）にそのまま適用すると、「部分適用された関数がFalse扱いされて破棄される」という実用上の致命的なバグを引き起こす。

ハードウェア（CPU）には圏論的真偽値は存在せず、「レジスタがゼロ(Null)か非ゼロか」の判定フラグ（AArch64の `CBZ/CBNZ` 等）のみが存在する。これに倣い、コンパイラ内部の判定ロジックを以下の2つに分離する。

1. **論理的真偽値 (`$is_truthy`)**:
   - 用途: 論理NOT (`!`) などの純粋な真偽値評価。
   - 挙動: Unit と クロージャを `False (0)` と判定する。（仕様通り）
2. **値の存在チェック / コントロールフロー (`$is_not_unit`)**:
   - 用途: `|` (ORフォールバック), `&` (AND連鎖), `:` (Guard分岐) などの処理継続判定。
   - 挙動: 純粋な Unit (値の不在) の場合のみ `False (0)` とし、クロージャを含む何らかの値が存在すれば `True (1)` としてアセンブリを分岐させる。

この物理的な存在チェックを導入することで、ポイントフリー・スタイルや関数合成における「関数の連鎖的受け渡し」が、オーバーヘッドなく（ハードウェアの分岐命令のまま）安全に実行可能となる。

## 静的ディスパッチと Staged Transformation（段階的変換）

Signの最大の特徴は「予約語がなく、記号の配置だけで型や文脈が決まる」という点にある。これを実現するため、Signのコンパイルプロセスでは、パーサの純粋性を維持しつつコンパイラで高度な最適化を行う「Staged Transformation」という設計思想を採用している。

### 1. パーサの純粋性とASTフェーズでの「救済（Rescue）」
Signのパーサは文脈依存の複雑な状態を持たず、数学的に純粋な構造を構築することに専念する。例えば、`[1 ~ 10]` のように中に式が1つしかないブラケットは、パーサによって即座に中身が取り出され（Unwrap）、フラットな `1 ~ 10` という式として扱われる。

しかし、この純粋な結合規則により、直感に反する優先順位の競合が発生する場合がある。
- **事例: Range演算子の結合救済**: 
  `0 ~+ 2 ~ 10` という記述は、素朴なパースでは右側の `~` が優先され `(0 ~+ (2 ~)) 10` と解釈されてしまう。
- **事例: インライン展開の巻き込み**: 
  `[+] [1 ~ 10]` は、関数適用の左結合性により `([+] 1 ~) 10` と解釈され、Rangeオブジェクトの生成途中でFold操作が割り込んでしまう。

コンパイラはパーサの文法定義を複雑にする（＝パーサを汚す）のではなく、**AST最適化フェーズ (`optimizeAst`) での木の組み替え（Rescue）** によってこれを解決する。コンパイラは中間状態のASTを静的に解析し、不完全なRange構造を検知すると、正しい結合順序へと木を再構成する。これにより、パーサのシンプルさと記述の直感性を両立させている。

### 2. Zero Cost Domain Abstraction（ゼロコストドメイン抽象化）
Signにおいて、リストやイテレータの操作は「抽象化のコスト」を支払うことなく実行される。これは、コンパイラが特定のパターンを検知してネイティブな計算構造に置換する「静的ディスパッチ」によって実現される。

- **インラインFold/Mapのインターセプト**:
  `[+]` や `[* 2,]` は、言語仕様上は「関数合成可能なラムダ式」へと自動展開されるが、これをそのまま実行するとクロージャの関数呼び出しオーバーヘッドが発生する。
  コンパイラは、Operator Section 特有のラムダ構造を AST フェーズで静的にパターンマッチして捕捉し、動的な関数呼び出しを完全に排除した **WASMネイティブな高速ループ（`inline_fold` / `inline_map`）** へとアセンブリレベルですり替える。

- **遅延評価としてのRange**:
  `[1 ~ 1000]` と記述しても、その時点ではメモリ上に1000要素のリストは構築されない。これは「開始・終了・ステップ」のみを保持する軽量なイテレータとして扱われる。このイテレータが `fold` や `map` に渡された瞬間、コンパイラは一度も中間リストを実体化させることなく、レジスタレベルのループ処理で結果を算出する。

人間が記述する「数学的に純粋な構造」が、実行時には「究極のパフォーマンスを持つ計算機械語」へと変換される。これがSignのセマンティクスの本質である。
