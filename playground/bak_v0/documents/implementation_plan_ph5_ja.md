# 実装計画: WASM フェーズ 5 (配列・文字列の展開とSVCストリームI/O)

## 目標
Sign言語の最もエレガントな機能の一つである「後置 `~`（展開演算子）」と「境界演算子 `#`」によるI/O処理を、WASM上で完全実装する。
当初想定していた「JSコンパイラによるASTの静的展開（インラインループ生成）」という命令型のアプローチを破棄し、圏論的裏付けに基づく**「タグ付け＆除法射ディスパッチ（SVCアーキテクチャ）」**としてランタイム側に実装を移行する。

## 現状の課題と解決策（アーキテクチャの刷新）
- **課題:** ASTパターンの網羅漏れや、カリー化された遅延評価オブジェクトとの相性の悪さにより、JS側での静的なループ展開ロジックが破綻（無限ループやNaN化）していた。
- **解決策:** コンパイラは単に「展開待ちフラグ」を立てるだけの軽量処理に徹し、実際の展開やI/O処理はWASMランタイム側が「型タグ」を見て動的にルーティングする（OSのシステムコール・SVC的アプローチ）。

## 実装した仕様と変更点

### 1. `~` (Expand / 展開演算子) の「タグ付け」システム
JSコンパイラ（`compiler_wat.js`）から巨大な `findUnroll` ロジックを完全削除。
コンパイラは `~` を検知すると、WASMランタイムの `$make_spread` システムコールを呼ぶ。
- ランタイムは元の型タグを検証し、**展開待ちタグ**に反転させる。
  - 通常のリスト (`0x7FFC`) -> スプレッドリスト (`0x7FF4`)
  - 通常の文字列 (`0x7FFD`) -> スプレッド文字列 (`0x7FF5`)
- 未定義変数などが展開されようとした場合は安全に `NaN` を返し、無限ループを防止する。

### 2. 境界演算子 `#` の SVC（スーパーバイザ・コール）化
WASM側の `$apply_hash` 関数を、型タグに基づく動的ディスパッチ（AArch64の `svc` 命令相当）として再実装。
左辺が `1.0`（標準出力）の場合、右辺の型タグによって以下の振る舞いへ分岐する。

- **ストリーム一括出力:** 通常の文字列 (`0x7FFD`) または スプレッド文字列 (`0x7FF5`) が来た場合、`~` の有無に関わらず `strlen` を計算し、環境側（JS）の `print_string` へ一括送信する。
- **構造体出力（シリアライズ）:** 展開されていないリスト (`0x7FFC`) やクロージャ (`0x7FFE`) が来た場合、環境側の `print_list` や `print_closure` へポインタのみを渡し、構造的表現を出力させる。
- **リストの動的展開:** スプレッドリスト (`0x7FF4`) が来た場合のみ、ランタイム内で動的にループを回し、要素を順次取り出して `#`（または空白適用）を実行する。タプル終端（`cdr` がポインタではなくスカラー値になるケース）も安全に検知してループを抜ける。

### 3. `#` のパススルー仕様（戻り値の透過）とストリームチェーン
副作用（I/O）と純粋関数の対称性を保つため、`#` 演算子の戻り値を一律の `NaN` から**「右辺の値のパススルー（そのまま返す）」**仕様に変更。
これにより、`(1 # list) | map_func` のような、値を出力しつつ次へ流すUnixの `tee` コマンド的な美しいストリームチェーンが可能となった。

## 期待される成果
- リストと関数の随伴関手定理に基づく、完全に対称的でバグのないWASM実行モデルの完成。
- コンパイラ（JS）の劇的な軽量化と、WASMバイナリサイズの縮小。
- 関数型言語としての「純粋なデータフロー構築」と「境界での一括I/O」という美しいプログラミングパラダイムの実現。
