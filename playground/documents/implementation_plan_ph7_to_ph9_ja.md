# 実装計画: WASM フェーズ 7〜9 (コンパイラ最適化・数学演算子・動的リスト構築)

## 目標
フェーズ6で「どんなコードでも正しく動く安全な動的ディスパッチ（最終防衛ライン）」が完成したことを受け、処理速度を極限まで高めるための「コンパイル時最適化（Optimizer）」を導入します。その後、高度な数学演算子や動的リスト構築の実装に進みます。

## ✅ フェーズ 7: コンパイル時最適化 (静的ディスパッチとβ簡約) 【完了】
設計思想である「アセンブリコードが長くなっても処理速度を優先する」方針に従い、WASM/AArch64の実行時オーバーヘッドを削ぎ落としました。

- **定数畳み込み (Constant Folding):** JS側で計算可能な式（算術・比較・論理・ビット演算）をコンパイル時に事前評価し、最速のネイティブ定数ロード（`f64.const`）に置換。Sign言語特有のTruthy/Falsy仕様（0.0も真）やビット反転仕様に完全準拠。
- **β簡約 (Beta Reduction / インライン展開):** `(x ? x + 10) 20` のような即時関数を検知し、クロージャ生成を省略して関数本体をインライン展開。
- **静的ディスパッチ:** 左辺が値である適用 `100 200` などをコンパイル時に検知し、動的ディスパッチ（`$apply_space`）をスキップして直接リスト構築（`$cons`）命令を出力。
- **タプルのスプレッド展開:** `f (10 20)` のようにタプル（値の連鎖）として渡された引数を、関数のカリー化に合わせて自動でフラットな引数としてスプレッド展開する仕様を確立。

## ✅ フェーズ 8: 単項数学演算子の完備 【完了】
- **絶対値 `|...|`**: ASTの `type: 'abs'` をWASMネイティブの `f64.abs` に直接バインディング。
- **前置マイナス `-`**: `f64.sub` ではなく、単項用の `f64.neg` を使用するように最適化。
- **階乗演算子 (後置 `!`)**: 再帰関数呼び出しではなく、WASM内部で `loop` と `block` 命令を用いた高速なネイティブ階乗ルーチンを出力。また、非数（`NaN`）が入力された際の無限ループを防ぐ安全設計（NaNガード）を実装。

## 🔄 フェーズ 9: 動的リスト構築 (Range / Continuous) 【次フェーズ】
- **Range演算子群 (`~`, `~+`, `~-` 等)**: 動的に長さが決まるリストをWASMヒープ上に高速に構築するループルーチンを出力します。
- **連続生成演算子 (前置 `~`)**: 遅延評価や無限リストの概念を取り入れたジェネレータクロージャ、または固定長メモリ領域の構築戦略を実装します。
