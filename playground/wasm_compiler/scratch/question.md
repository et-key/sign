`update.sn` の以下の記述について：
```sign
point :
	x
	y
	z
```
ここでは `x \n y \n z` というインデントブロックが「構造体（辞書型）」を生成しています。

一方で、前回の `match_case.sn` では：
```sign
func_match : x y ?
  x < 1 : 'result1'
  y < 3 : 'result2'
```
ここでは `\n` が「前の行が Unit (解なし) なら次の行を評価する」というフォールバック（Maybeモナドの短絡評価）として機能しています。

**質問:**
Signの言語仕様として、コンパイラは `\n` の振る舞い（フォールバックになるか、構造体の要素としてメモリに並べるか）をどのように区別すべきでしょうか？

1. **型（戻り値）による区別:** 
   各行が `Unit` を返す可能性がある（`:` の条件式など）場合はフォールバックとして扱い、必ず値を持つ（識別子や定数など）場合は構造体アロケーションとして扱う。
2. **カッコの有無による区別:**
   関数本体のトップレベルインデントはフォールバックとして働き、構造体を作りたい場合は**必ず `[` `]` で囲む**必要がある。（例: `point : [ \n x \n y \n z \n ]`）
3. **その他の規約（セマンティクス）:**
   実は `\n` はすべて構造体を生成しており、`match_case` は「条件付きの要素を持つ構造体」を生成し、その構造体を評価した結果が単一の値になる？

実装上、非常に重要なコアロジックとなるため、想定されている評価戦略を教えていただけますでしょうか。
